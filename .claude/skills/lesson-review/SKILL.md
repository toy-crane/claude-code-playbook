# lesson-review

강의 사이트(`content/docs/**/*.mdx`) 레슨 페이지를 **독자의 스캔 가독성** 관점에서 리뷰하는 스킬. heading 구조가 TOC 에서 어떻게 보이고, 독자가 페이지를 처음 훑을 때 이해 흐름이 자연스러운지를 점검한다.

## 입력

argument 로 파일 또는 폴더 경로를 받는다.

- 폴더: 내부 `*.mdx` 를 재귀적으로 찾아 **파일별로 하나씩 대화형 리뷰** (일괄 처리 금지).
- `public: false` 파일도 **포함** (편집 중일 가능성이 높음). 단 리뷰 결과에 `public: false` 임을 명시.

## 처리 흐름 (파일당)

1. Read → heading 트리(H1~H3) 요약 + 카테고리별 룰 스캔.
2. 성격이 서로 다른 **개선안 3가지** 제시 + 그 중 **하나를 추천안으로 지목**.
3. 사용자 응답 처리:
   - **A/B/C 선택** → Edit 반영 → 결과 요약.
   - **수정 요청** → 피드백 받아 제안 재생성.
   - **건너뛰기** → 그대로 두고 다음 파일로.
4. 폴더 인자면 한 파일 완결 후 다음으로. 마지막에 **처리 요약** (건너뛴 / 적용한 / 이슈 없던 파일) 출력.

## 카테고리 1 — 구조 & 네비게이션

독자가 페이지를 처음 스캔할 때 TOC / heading 계층 / 순서가 자연스러운지 **판단**. 정량 임계값 없음. 문맥으로 가늠.

### 룰 1 — TOC 가치 검증

한 가지 질문: **"이 heading 이 목차에 있어서 독자가 얻는 네비게이션 가치가 무엇인가?"**

가치가 낮거나 없는 heading 은 flag 대상. 전형적인 패턴:

- **반복 heading** — 동일/거의 동일한 이름이 같은 페이지에 여러 번 (예: `macOS`, `Windows`, `설치 확인` 이 각 도구마다 반복). TOC 만 봐선 "몇 번째 macOS 인가" 구분 불가.
- **체재 분기 heading** — "OS 별", "플랫폼 별" 같은 분기용 heading. 본문 흐름엔 필요해 보이지만 TOC 에선 랜드마크 역할을 못 함.
- **지나치게 일반적인 단어** — `개요`, `설명`, `정리`, `설치`, `사용법` 등만 덩그러니. 어떤 섹션인지 문맥이 안 나옴.
- **본문만 봐도 파악되는 heading** — 제거해도 독자가 정보 위치를 놓치지 않는 경우.

권장 조치: **bold 로 강등** (`**macOS**`), **제거**, 혹은 **상위 섹션에 흡수/통합**.

### 룰 2 — 그룹핑 가독성

한 가지 질문: **"독자가 처음 스캔할 때 어떤 계층 / 순서 / 묶음으로 보면 이해가 가장 쉬울까?"**

heading level, 순서, 형제 관계, 의미 경계를 전체적으로 본다. 전형적인 패턴:

- **의미가 다른 형제 H2** — 성격이 다른 섹션이 같은 H2 레벨로 나란히 (예: "1~8번 도구 설치" + 그 뒤의 "최종 체크리스트"). 제목 재명명, 레벨 조정, `---` 강화, 또는 분리 고려.
- **근접 H2** — 두 H2 사이 본문이 너무 짧아 TOC 가 둘 다 active 로 표시. 앞 섹션을 확장하거나 둘을 합치거나 계층을 바꿈.
- **순서 문제** — 난이도/의존성/독자 여정과 맞지 않는 순서.
- **묶음 누락** — 의미상 묶여야 할 것들이 평평하게 나열됨.
- **과도한 계층** — H4 이하로 깊이 들어가 독자가 길을 잃음.

분석은 "이 레슨의 독자가 누구인지" (수강생/초보/숙련자) 와 "무엇을 알고 싶어 이 페이지에 왔는지" 를 가정하고 진행.

## 카테고리 2 — Fumadocs 포맷 린트

렌더링이 깨지지 않게 지키는 **기계적 계약**. 판단 여지 없음. 어기면 flag.

### 룰 3 — 내부 링크는 절대 경로

MDX 간 내부 링크는 `/docs/<slug>` 절대 경로 사용. `./` / `../` 로 시작하는 **비-이미지** 링크는 flag — 루트 `/docs` 페이지에서 브라우저 URL 해석상 404.

권장 조치: `/docs/<slug>` 로 변환. 이미지만 예외 — `![alt](./attachments/foo.png)` 유지.

### 룰 4 — 본문 H1 금지

Fumadocs 가 frontmatter `title` 로 페이지 상단 `<h1>` 을 자동 렌더한다. 본문에 `# 제목` 을 다시 쓰면 H1 이 두 번 나와 시각적/SEO 양쪽으로 깨진다.

- 본문 어디든 `# ` 으로 시작하는 줄이 있으면 flag.
- 권장 조치: 해당 줄 제거. 본문 heading 은 `##` 레벨부터. 예외 없음.

### 룰 5 — `<Callout>` JSX 사용 (markdown 콜아웃 금지)

`> [!NOTE|TIP|WARNING|INFO|...]` 같은 GFM/Obsidian 콜아웃 문법은 **금지**. 현재 Fumadocs 설정에 GFM alert 플러그인이 없어 `[!TYPE]` 이 리터럴 텍스트로 노출된다.

- `> [!...]` 패턴이 파일 내에 존재하면 모두 flag.
- 권장 조치: Fumadocs `<Callout>` JSX 로 변환.
- 타입 매핑: `[!NOTE]`, `[!INFO]` → `type="info"` / `[!TIP]` → `type="idea"` / `[!WARNING]` → `type="warn"`.
- 변환 예시:
  ```mdx
  <Callout type="info" title="단어? 토큰?">
  본문 내용…
  </Callout>
  ```
- 제목에 inline code (``` ` ```) 나 markdown 강조가 필요하면, `title` 대신 본문 첫 줄에 bold 로 적는다 (JSX 속성은 markdown 을 렌더하지 않음).
- `<Callout>` 은 `src/components/mdx.tsx` 통해 전역 등록 → 파일별 import 불필요.
- 유효 `type`: `info | warn | error | success | warning | idea`.

## 카테고리 3 — 본문 스타일

문단 묶음과 말투 같은 **문장 수준 글쓰기 규칙**. 취향 영역이지만 레슨 간 톤을 유지하기 위해 일관되게 적용.

### 룰 6 — 문단은 의미 단위로 묶기

빈 줄 = `<p>` 경계. Fumadocs(Tailwind prose) 가 `<p>` 마다 margin 을 붙이므로 문장마다 빈 줄이 있으면 간격이 과도해진다.

판단 질문: **"같은 논점을 설명하는 문장들이 같은 문단에 묶여 있나?"**

- 같은 주제 연속 문장 → 한 문단에 이어 쓰기 (빈 줄 없음).
- 주제 전환 → 빈 줄로 분리.
- 강조용 standalone 단문은 페이지당 1~2개까지 OK.

Flag 대상:

- 한 섹션에서 3문장 이상이 **전부 빈 줄로 분리** (1문장 1문단 패턴).
- 10줄 초과 문단에 논점 2~3개가 섞임.

권장 조치: 같은 주제 문장을 한 문단으로 합치기. 긴 문단은 논점 전환 지점에서 분리.

### 룰 7 — 합니다체 prose 말투

레슨의 **narrative prose 전체**는 **합니다체** (`~합니다`, `~입니다`) 로 쓴다. 독자에게 말을 거는 모든 문장이 같은 어조를 유지해야 제목 → subtitle → 본문 → 콜아웃 사이 톤이 끊기지 않는다.

**Flag 대상** — 아래 영역에 `~한다`, `~된다`, `~이다`, `~아니다`, `~없다`, `~배운다`, `~이해한다` 등 해라체 종결이 나오면 flag:

- frontmatter 최상위 `description` 값
- 단락 본문
- 콜아웃(`<Callout>`) 내부 서술 문장
- 불릿 리스트 중 "완전한 서술 문장" 성격의 항목 (학습 목표, 설명 목록 등)

**Flag 제외** — 의도적 스타일 / 형식 모사 / 인용 맥락:

- **Heading 태그라인** — `## Spec이 나머지를 결정한다` 같이 heading 전체가 짧은 문장형 제목인 경우. heading 은 강조를 위해 해라체 허용.
- **frontmatter `title`** — 제목도 해라체 태그라인 허용 (예: `title: '위임 범위가 커지면, 경계도 단단해져야 한다 | SDD'`).
- **코드 펜스 안 내용** — 예시 리터럴 (CLAUDE.md / skill frontmatter / rule 파일 등) 이므로 변경 금지.
- **블록쿼트 안 인용** — `> ...` 로 감싼 인용문은 원본 유지.
- **기능 요구사항 / 수용 기준 리스트** — "사용자가 X 하면, Y 된다" 형식의 명세 리스트는 형식 모사이므로 해라체 유지.

조치: 해라체 → 합니다체. 명사 종결 subtitle (`~안내`, `~가이드`) 은 동사형 합니다체로 재작성 (`~안내합니다`).

## 출력 포맷

```
## 📄 <파일 경로>

### 현재 heading 트리
<H2/H3 요약 — 들여쓰기로 계층 표시>

### 발견된 이슈
- <이슈 1 — 룰 번호 명시>
- <이슈 2 — 룰 번호 명시>
...

### 개선안

#### A안 — <한 줄 요약>
- 변경: <구체적으로 무엇을 바꾸는가>
- 효과: <독자/TOC 관점에서 어떤 개선이 있나>
- 트레이드오프: <잃는 것 / 주의점>

#### B안 — <한 줄 요약>
...

#### C안 — <한 줄 요약>
...

### 추천: <A | B | C>안

<왜 이 안이 이 레슨에 가장 적합한가 — 독자/의도/트레이드오프 균형 근거 1~2문장>

어떤 안으로 진행할까요? (A / B / C / 수정 요청 / 건너뛰기)
```

### 개선안 3가지 구성 원칙

세 안은 **성격이 서로 달라야** 한다. 복붙 변형 금지. 아래 중 한 축으로 스펙트럼 구성:

- **변경 강도**: 최소 변경(A) ↔ 절충(B) ↔ 구조 재편(C)
- **관점**: TOC 네비게이션 / 스캔 가독성 / 의미 계층
- **독자 우선순위**: 초보 / 숙련자 / 절충

각 안에는 **트레이드오프** 명시 — 완벽한 안은 없다.

### 추천안 선정 기준

세 안 중 하나를 추천으로 지목. 종합 판단 기준:

- **독자 적합성** — 레슨의 주 독자(초보/숙련자) 와 방문 의도에 가장 맞는가.
- **트레이드오프 균형** — 잃는 것 대비 얻는 것이 가장 큰가.
- **현재 상태와의 거리** — 잘 쓰인 레슨이면 최소 변경, 구조가 크게 어긋났으면 재편.

추천은 **기본값 제안일 뿐 강제 아님**. 다른 두 안도 동일 품질로 제시. 사용자가 명시적으로 선택할 때만 Edit.

## 하지 않는 것 (Out of scope)

- 이미지 alt 텍스트 / 경로 / 존재 여부
- 맞춤법 (말투/어조는 룰 7)
