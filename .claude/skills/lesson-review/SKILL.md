# lesson-review

강의 사이트(`content/docs/**/*.mdx`)의 레슨 페이지를 **독자의 스캔 가독성** 관점에서 리뷰하는 스킬. heading 구조가 TOC 에서 어떻게 보이고, 독자가 페이지를 처음 훑을 때 이해 흐름이 자연스러운지를 점검한다.

## 언제 쓰는가

- 강의 페이지를 편집 중이거나 신규 레슨을 막 작성한 직후, 사용자가 명시적으로 호출할 때.
- **커밋 전 자동 게이트 아님.** 자동으로 돌리지 말 것.

## 입력

argument 로 파일 또는 폴더 경로를 받는다.

- 파일: 해당 MDX 한 개만 리뷰
- 폴더: 내부 `*.mdx` 를 재귀적으로 찾아 **파일별로 하나씩 대화형 리뷰**
- `public: false` 파일도 **포함**한다 (편집 중 상태일 가능성이 높음). 단 리뷰 결과 안내에 `public: false` 라는 점을 명시.

argument 가 없으면 사용자에게 경로를 물어본다.

## 처리 흐름 (파일 하나당)

1. 파일을 Read 한다.
2. heading 트리 (H1~H3) 를 추출해 요약한다.
3. 아래 두 룰로 이슈를 찾는다.
4. **서로 다른 성격의 개선안 3가지**를 제시한다.
5. 사용자가 안을 선택하면 Edit 로 반영한다.
6. 폴더 인자인 경우 다음 파일로 넘어간다. 각 파일은 **완전히 끝낸 뒤** 다음 파일을 시작 (일괄 처리 금지).

## 룰 1 — TOC 가치 검증 (포괄)

한 가지 질문으로 판단: **"이 heading 이 목차에 있어서 독자가 얻는 네비게이션 가치가 무엇인가?"**

가치가 낮거나 없는 heading 은 flag 대상이다. 구체적 수치 기준은 두지 않고 문맥을 보고 판단한다. 전형적인 패턴은 다음과 같다.

- **반복 heading** — 동일 또는 거의 동일한 이름의 heading 이 같은 페이지에 여러 번 등장 (예: `macOS`, `Windows`, `설치 확인` 이 각 도구마다 반복). TOC 만 봐서는 "몇 번째 macOS 인가" 구분이 안 되므로 네비게이션 가치 0 에 가까움.
- **체재 분기 heading** — 의미 단위가 아니라 "OS 별", "플랫폼 별", "언어 별" 같은 분기용 heading. 본문 흐름에선 필요해 보이지만 TOC 에선 랜드마크 역할을 못 함.
- **지나치게 일반적인 단어** — `개요`, `설명`, `정리`, `설치`, `사용법` 등만 덩그러니 — 어떤 섹션인지 문맥이 안 나옴.
- **본문만 봐도 바로 파악되는 heading** — heading 을 제거해도 독자가 정보 위치를 놓치지 않는 경우.

권장 조치: **bold 텍스트로 강등** (`**macOS**`), **제거**, 혹은 **상위 섹션에 흡수/통합**.

## 룰 2 — 그룹핑 가독성 분석 (포괄)

한 가지 질문으로 판단: **"독자가 이 페이지를 처음 스캔할 때 어떤 계층 / 순서 / 묶음으로 보면 이해가 가장 쉬울까?"**

heading level, 순서, 형제 관계, 의미 경계를 전체적으로 본다. 정량 임계값은 없다. 전형적인 패턴은 다음과 같다.

- **의미가 다른 형제 H2** — 성격이 다른 섹션이 같은 H2 레벨로 나란히 있어 TOC 에서 한 덩어리로 보임 (예: "1~8번 도구 설치" + 그 뒤의 "최종 체크리스트"). 제목 재명명, 레벨 조정, `---` 강화, 또는 분리 고려.
- **근접 H2** — 두 H2 사이 본문이 너무 짧아 뷰포트에 함께 들어와 TOC 가 둘 다 active 로 표시되는 경우. 앞 섹션을 확장하거나 둘을 합치거나 계층을 바꿈.
- **순서 문제** — 난이도/의존성/독자 여정과 맞지 않는 순서.
- **묶음 누락** — 의미상 묶여야 할 것들이 평평하게 나열됨.
- **과도한 계층** — H4 이하로 깊이 들어가 독자가 길을 잃음.

분석은 "이 레슨의 독자가 누구인지" (수강생, 초보, 숙련자) 와 "무엇을 알고 싶어 이 페이지에 왔는지" 를 가정하고 진행한다.

## 출력 포맷

리뷰 결과는 아래 템플릿을 따른다.

```
## 📄 <파일 경로>

### 현재 heading 트리
<H2/H3 요약 — 들여쓰기로 계층 표시>

### 발견된 이슈
- <이슈 1 — 룰 번호 명시>
- <이슈 2 — 룰 번호 명시>
...

### 개선안

#### A안 — <한 줄 요약>
- 변경: <구체적으로 무엇을 바꾸는가>
- 효과: <독자/TOC 관점에서 어떤 개선이 있나>
- 트레이드오프: <잃는 것 / 주의점>

#### B안 — <한 줄 요약>
...

#### C안 — <한 줄 요약>
...

어떤 안으로 진행할까요? (A / B / C / 수정 요청 / 건너뛰기)
```

### 개선안 3가지 구성 원칙

세 안은 **성격이 서로 달라야** 한다. 복붙 변형을 피한다. 다음 중 한 축을 골라 스펙트럼을 구성한다.

- **변경 강도**: 최소 변경(A) ↔ 절충(B) ↔ 구조 재편(C)
- **관점**: TOC 네비게이션 위주(A) / 스캔 가독성 위주(B) / 의미 계층 위주(C)
- **독자 우선순위**: 초보 우선(A) / 숙련자 우선(B) / 양쪽 절충(C)

각 안에는 항상 **트레이드오프**를 명시한다 — 완벽한 안은 없다.

## 승인 & 적용

- 사용자가 A/B/C 중 선택 → Edit 로 반영 → 반영 결과 요약.
- 사용자가 "수정 요청" → 피드백 받아 제안 재생성.
- 사용자가 "건너뛰기" → 해당 파일은 그대로 두고 다음 파일로.
- 폴더 인자일 때 모든 파일 처리가 끝나면 **처리 요약** (건너뛴 파일 / 적용한 파일 / 이슈 없던 파일) 을 마지막에 출력.

## 하지 않는 것 (Out of scope)

다음 품질 체크는 이 스킬이 하지 않는다. 추후 다른 스킬에서 다룰 것.

- 이미지 alt 텍스트 / 경로 / 존재 여부
- 위키링크 / 마크다운 링크 깨짐 여부
- 콜아웃 스타일 일관성
- 코드 블록 언어 지정 / 포맷
- 맞춤법 / 문체 / 어조
- frontmatter 스키마 검증

## 참고 — 오늘 실제로 적용한 사례

본 프로젝트 `content/docs/prerequisites.mdx` 에서:

- **룰 1 적용**: 8개 도구마다 반복되던 `macOS` / `Windows` / `설치 확인` H3 (총 18개) 를 bold 텍스트로 강등. 권장 조치의 전형 케이스.
- **룰 2 적용**: `## 8. Discord` 와 `## 설치 확인 체크리스트` 가 같은 H2 레벨이라 TOC 에서 형제로 보이고, 뷰포트에 함께 잡혀 둘 다 active 로 강조되던 문제. 제목을 `## 최종 체크리스트` 로 재명명해 "1~8번 도구 설치" 와 성격이 다르다는 걸 드러냄.

새 리뷰에서도 이 수준의 구체성으로 제안할 것.
