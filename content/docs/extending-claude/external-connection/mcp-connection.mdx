## Overview

앞 레슨에서 gh CLI 로 Claude 가 실시간 GitHub 데이터에 닿도록 만들었습니다. 그럼 궁금해집니다 — CLI 로 이렇게 많은 것이 되는데, **MCP 는 언제 필요한가?**

이번 레슨은 CLI 대세가 된 지금도 MCP 가 여전히 더 나은 상황이 어디인지 세 가지로 정리하고, **Chrome DevTools MCP** 로 Claude 에게 실제 브라우저를 조종하는 능력을 주는 실습을 합니다.

### 학습 목표

- MCP 가 CLI 로 대체되기 어려운 세 가지 상황을 설명할 수 있습니다
- MCP 의 Server · Client · Host 구조를 이해합니다
- Chrome DevTools MCP 를 설치하고 Claude 가 실제 페이지를 관찰하게 만듭니다

### 시작하기 전 확인사항

- Node.js 20.19 이상이 설치돼 있어야 합니다 (`node --version`)
- Chrome 현재 안정 버전이 설치돼 있어야 합니다
- 실습 프로젝트 시작 브랜치로 전환합니다 — `git checkout ch07-02`

## CLI 가 다 되는데 MCP 를 배워야 할까?

gh·aws·kubectl·supabase — 개발자가 쓰는 거의 모든 외부 서비스에 CLI 가 있고, Claude 는 그걸 바로 씁니다. 그러면 MCP 는 왜 남아 있을까요?

문제는 CLI 가 도달하지 못하는 영역입니다. "이 페이지 로딩 이상한데 한번 봐줘" 라는 요청을 떠올려 봅니다. CLI 로 서버에 HTTP 요청을 보낼 수는 있어도, **브라우저가 JS 를 실행한 뒤의 DOM · 콘솔 에러 · 네트워크 타이밍** 은 CLI 로는 관찰할 수 없습니다. JS 가 안 돌아간 상태의 HTML 을 받아오는 것과, 렌더링이 끝난 실제 화면을 보는 것은 다른 세계입니다.

이런 영역에는 **다른 종류의 연결** 이 필요합니다.

## MCP 가 여전히 더 나은 세 가지 상황

<McpThreeStrengths />

MCP 는 AI 가 외부 시스템을 **Tool** 로 쓰게 해주는 표준 프로토콜입니다. MCP 서버 하나를 붙이면, 그 서버가 노출하는 Tool 들을 Claude 가 내장 도구처럼 호출합니다. CLI 가 있어도 MCP 가 여전히 더 나은 상황은 세 가지입니다.

### ① 실행 중인 앱 상태 관찰

Chrome 의 렌더링된 DOM, Playwright 가 구동한 페이지의 콘솔, Figma 에서 열려 있는 디자인 파일 — 전부 **실행 중인 앱 내부의 실시간 상태** 입니다. CLI 는 프로세스 밖에서 HTTP 로 요청할 뿐, 앱 내부 상태에는 닿지 못합니다.

MCP 서버는 이런 앱 위에서 직접 돌면서 내부 상태를 Tool 로 노출합니다. 이 레슨의 실습이 이 유형입니다.

### ② OAuth 인증이 복잡한 외부 서비스

Linear·Notion·Sentry·Supabase 같은 SaaS 는 CLI 도 있지만, CLI 인증에 OAuth 플로우·토큰 발급·갱신이 따라옵니다. **MCP 서버는 OAuth 플로우를 내장** 해서, 설치한 뒤 브라우저 한 번 승인하면 인증이 끝납니다. 팀원이 같은 MCP 를 쓸 때 인증 설정을 반복 설명하지 않아도 됩니다.

### ③ 팀 공유 설정

`.mcp.json` 파일에 MCP 설정을 넣고 커밋하면, **팀원 누구든 저장소를 받는 순간 같은 MCP 를 같은 설정으로 사용** 합니다. CLI 는 각자 `brew install`·`PATH`·`alias` 를 맞춰야 하지만, MCP 는 파일 하나가 설정의 단일 출처입니다.

<Callout type="info" title="어떤 것을 실습에서 다루는가">
이 레슨은 ① (실행 중인 앱 상태 관찰) 을 Chrome DevTools MCP 로 체험합니다. ②·③ 도 같은 원리로 동작하므로, 이 레슨이 끝나면 Linear·Notion 같은 MCP 도 거의 동일한 방식으로 붙일 수 있습니다.
</Callout>

## MCP 의 동작 방식: Server · Client · Host

<McpArchitecture />

MCP 는 세 개의 역할로 나뉩니다.

- **MCP Server** — 외부 시스템을 감싸서 Tool 로 노출하는 프로세스. Chrome DevTools MCP 서버는 내부적으로 Chrome 을 띄워서 그 상태를 Tool 로 제공합니다
- **MCP Client** — Server 에 연결해서 Tool 을 호출하는 쪽. Claude Code 가 이 역할
- **MCP Host** — Client 를 품고 있는 사용자 프로그램. Claude Code 자체가 Host 이기도 합니다. Claude Desktop·Cursor·Windsurf 도 각각 Host

Claude Code 는 시작될 때 등록된 MCP 서버 프로세스를 자동으로 띄우고, 표준 입출력 (stdio) 으로 JSON 메시지를 주고받습니다. **사용자 입장에서는 내장 Tool 과 MCP Tool 의 차이를 느끼지 못합니다** — 똑같이 "Claude 가 쓸 수 있는 도구" 로 보입니다.

## [실습] Chrome DevTools MCP 로 페이지 관찰하기

이제 Claude 가 실제 브라우저를 조종하도록 MCP 를 붙입니다.

### Step 1: MCP 서버 등록

프로젝트 루트에서 다음 명령어를 실행합니다.

```shell
claude mcp add chrome-devtools -s project -- npx -y chrome-devtools-mcp@latest
```

| 부분 | 의미 |
|------|------|
| `chrome-devtools` | MCP 서버 이름. `/mcp` 에서 이 이름으로 표시됩니다 |
| `-s project` | 스코프. 현재 프로젝트의 `.mcp.json` 에 저장됩니다 |
| `--` | 구분자. 이 뒤는 서버 실행 명령어 |
| `npx -y chrome-devtools-mcp@latest` | npm 에서 공식 패키지를 받아 실행합니다 |

<Callout type="info" title="-s project 를 쓰는 이유">
`-s user` 로 설치하면 모든 프로젝트에서 이 MCP 가 로드되고 매 프로젝트에 영향을 줍니다. `-s project` 는 `.mcp.json` 파일에만 기록돼서, 이 레슨 프로젝트에서만 MCP 가 살고 다른 작업에는 간섭하지 않습니다.
</Callout>

### Step 2: Claude Code 재시작

MCP 설정은 Claude Code 가 **시작될 때 한 번** 읽습니다. 기존 세션에서는 반영되지 않으므로 `/exit` 후 다시 `claude` 를 실행합니다.

### Step 3: 연결 확인

```plain text
/mcp
```

`chrome-devtools` 서버가 "Connected" 상태로 표시되면 성공입니다. 첫 실행 시 Chrome 을 다운받으므로 몇 초~몇십 초가 걸릴 수 있습니다.

### Step 4: 페이지 열고 콘솔 확인하기

이제 Claude 에게 특정 페이지를 열어서 콘솔을 봐달라고 요청합니다.

```plain text
https://www.google.com 을 열어서 콘솔 메시지가 있는지 확인해줘
```

Claude 는 MCP 의 Tool 을 순서대로 호출합니다.

1. `navigate_page` — 지정한 URL 을 엽니다
2. `list_console_messages` — 열린 페이지의 콘솔 메시지를 수집합니다

MCP 서버가 내부적으로 Chrome 을 띄우고, Chrome DevTools Protocol 로 DOM·콘솔·네트워크에 접근합니다. **CLI 로 `curl https://google.com` 을 돌리면 HTML 원본만 얻지만, MCP 로는 JS 가 실행된 뒤의 콘솔까지 볼 수 있습니다.**

### Step 5: 스크린샷과 네트워크 요청 확인

좀 더 깊이 들여다봅니다.

```plain text
같은 페이지에서 스크린샷을 찍고, 네트워크 요청 목록도 확인해줘
```

Claude 가 호출하는 Tool 이 바뀝니다.

- `take_screenshot` — 현재 페이지의 이미지
- `list_network_requests` — 페이지가 보낸 HTTP 요청 목록

**CLI 는 "서버에 무엇을 요청했는가" 는 내가 직접 보내야 알지만, MCP 는 브라우저가 자동으로 보낸 요청들을 전부 관찰할 수 있습니다.**

## MCP 를 연결할 때 치르는 비용

MCP 에도 대가가 있습니다. 서버 하나를 연결하면 그 서버가 제공하는 **모든 Tool 의 설명이 Context Window 에 로드** 됩니다. Chrome DevTools MCP 는 33 개 Tool 을 제공하므로, 설명만으로도 수천 토큰이 소비됩니다.

그래서 기본 원칙은 **"현재 작업에 필요한 MCP 만 연결한다"** 입니다. `.mcp.json` 을 프로젝트별로 관리하면, 이 프로젝트엔 Chrome DevTools 만, 다른 프로젝트엔 Linear 만 로드되도록 깔끔하게 분리됩니다.

CLI 는 실행할 때만 결과가 들어오므로 이 비용이 없습니다. 그래서 **CLI 로 해결되는 일엔 CLI, MCP 가 꼭 필요한 영역엔 MCP** 가 기본 전략입니다.

## 핵심 포인트 정리

1. **MCP 는 CLI 가 못 닿는 영역에 쓰입니다**: 실행 중 앱 상태 관찰 / OAuth 가 복잡한 서비스 / 팀 공유 설정 세 가지 상황에서 MCP 가 여전히 더 나은 선택입니다
2. **Chrome DevTools MCP 는 실제 Chrome 을 띄워 렌더링된 상태를 관찰합니다**: `navigate_page`·`list_console_messages`·`take_screenshot`·`list_network_requests` 같은 Tool 로 JS 가 실행된 뒤의 DOM·콘솔·네트워크를 Claude 가 직접 볼 수 있습니다
3. **MCP 는 Context 를 소비하므로 프로젝트에 필요한 것만 연결**: `-s project` 로 `.mcp.json` 에 저장하면 프로젝트별로 필요한 MCP 만 선별할 수 있습니다

## FAQ

- **Q: 이미 열려 있는 제 크롬 창에 붙일 수는 없나요?**
  - A: 기본값은 MCP 서버가 새 Chrome 프로세스를 띄우는 방식입니다. 기존 Chrome 에 붙이고 싶다면 Chrome 을 `--remote-debugging-port=9222` 로 시작하고, MCP 설정에 `--browser-url=http://127.0.0.1:9222` 옵션을 추가하면 됩니다. 대부분의 실습에선 새 Chrome 을 띄우는 기본값이면 충분합니다

- **Q: MCP 를 많이 붙이면 Claude 가 더 똑똑해지나요?**
  - A: 아닙니다. MCP 는 **접근 범위** 를 넓힐 뿐, 추론 능력을 향상시키지 않습니다. 오히려 Tool 설명이 Context 를 차지해서 성능이 떨어질 수 있습니다. 현재 작업에 쓰는 MCP 만 연결하는 것이 좋습니다

- **Q: MCP 서버를 별도로 실행해야 하나요?**
  - A: 아닙니다. `claude mcp add` 로 등록하면 Claude Code 가 시작될 때 서버 프로세스를 자동으로 띄우고, 종료될 때 같이 정리합니다. 별도 관리가 필요하지 않습니다

- **Q: MCP 는 Claude Code 전용인가요?**
  - A: 아닙니다. MCP 는 오픈 표준이므로 Claude Desktop·Cursor·Windsurf 등 MCP 를 지원하는 다른 AI 도구에서도 같은 MCP 서버를 재사용할 수 있습니다
