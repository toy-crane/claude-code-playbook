## Overview

앞 세 레슨에서 Claude 에게 외부 시스템 접근을 세 가지 방식으로 열었습니다 — gh CLI, Chrome DevTools MCP, 직접 만든 날씨 MCP. Claude 는 이제 이슈를 조회하고, 페이지를 관찰하고, 사내 API 를 호출할 수 있습니다.

그런데 실무에서 같은 작업을 시킬 때마다 결과가 다릅니다. "이 PR 리뷰해줘" 가 때로는 깊게, 때로는 대충, 때로는 포맷이 들쭉날쭉합니다. **접근할 수 있는 것** 과 **매번 같은 방식으로 쓰는 것** 은 다른 문제입니다.

이번 레슨은 외부 도구 (CLI · MCP) 에 Skill 을 결합해서 반복 워크플로우를 일관되게 만드는 원리와 두 가지 사례를 배웁니다.

### 학습 목표

- 도구 (Capability) 와 Skill (Procedure) 의 역할 차이를 구분합니다
- 외부 도구 + Skill 결합이 어떻게 일관성을 만드는지 이해합니다
- CLI+Skill, MCP+Skill 각각의 실전 구성을 설계할 수 있습니다

## 도구는 할 수 있게, Skill 은 매번 같게

<CapabilityVsProcedure />

망가진 캐비닛을 고치러 철물점에 갔다고 해봅니다. 작업대 위에는 접착제·클램프·경첩·전동 드릴이 전부 준비돼 있습니다. 도구가 어디서 왔는지는 중요하지 않습니다 — 매대에서 새로 사 온 것이든, 집에서 가져온 것이든 **작업대 위에 올라오는 순간 그냥 도구** 입니다.

그런데 도구가 아무리 많아도, 어떤 순서로 써야 하는지 모르면 캐비닛은 고쳐지지 않습니다. 옆에 있는 숙련된 직원이 순서를 안내해야 비로소 작업이 시작됩니다.

이것이 이 레슨의 핵심 구분입니다.

- **도구 (Capability Layer)** — CLI 와 MCP. "Claude 가 **무엇에 닿을 수 있는가**" 를 결정합니다. gh CLI 가 있으면 GitHub 에 닿고, Chrome DevTools MCP 가 있으면 브라우저에 닿습니다
- **절차 (Procedure Layer)** — Skill. "그 도구를 **어떤 순서로 · 어떤 기준으로 · 어떤 형식으로** 쓸지" 를 정의합니다

도구 없이는 일을 시작할 수 없고, 절차 없이는 매번 결과가 달라집니다.

## CLI + Skill 로 코드 리뷰 매번 같은 기준으로

<CombinedWorkflow />

`gh` CLI 로 PR diff 를 가져오는 것 자체는 한 줄이면 됩니다. 그런데 "이 PR 리뷰해줘" 라고만 하면 매번 리뷰 범위 · 형식 · 심각도 기준이 달라집니다. 여기에 **code-review Skill** 을 붙이면 달라집니다.

**역할 분담:**

- **gh CLI (Capability)** — PR diff 조회, 파일 목록 확인, 리뷰 코멘트 등록
- **code-review Skill (Procedure)** — 검사 항목 (네이밍 · 에러 처리 · 테스트 커버리지), 리뷰 형식, 심각도 기준 정의

**돌아가는 순서:**

1. Skill 이 리뷰 대상 PR 을 확인하고 팀 검사 항목을 로드 (탐색)
2. CLI 가 `gh pr diff` 로 변경 사항을 가져옴 (접근)
3. Skill 이 팀 컨벤션 기준으로 코드를 분석하고 피드백을 구조화 (조율)
4. CLI 가 `gh pr review` 로 구조화된 리뷰 코멘트를 등록 (실행)

Skill 없이 CLI 만 있으면 diff 는 볼 수 있지만 **무엇을 기준으로 리뷰할지** 가 매번 달라집니다. CLI 없이 Skill 만 있으면 기준은 완벽해도 PR 데이터에 닿지 못합니다. **CLI + Skill 결합** 이 있어야 "같은 기준으로 매번 같은 형식의 리뷰" 가 나옵니다.

## MCP + Skill 로 UI 버그 리포트 매번 같은 형식으로

두 번째 사례는 MCP 를 씁니다. Chrome DevTools MCP 로 브라우저 안을 볼 수 있게 됐지만, "이 페이지 이상해" 라고만 하면 Claude 는 감으로 몇 가지를 확인할 뿐입니다. **ui-bug-report Skill** 을 붙이면 매번 같은 형식의 리포트가 나옵니다.

**역할 분담:**

- **Chrome DevTools MCP (Capability)** — 현재 페이지의 콘솔 · 네트워크 · DOM · 스크린샷 수집
- **ui-bug-report Skill (Procedure)** — 심각도 분류 기준, 리포트 템플릿, 재현 스텝 형식 정의

**돌아가는 순서:**

1. MCP 가 페이지를 열고 `list_console_messages`·`list_network_requests` 로 에러 · 실패 요청 수집 (접근)
2. Skill 이 critical / major / minor 기준으로 분류 (조율)
3. Skill 이 재현 스텝 · 환경 정보 · 영향 범위를 팀 버그 리포트 템플릿에 채움 (조율)
4. MCP 가 `take_screenshot` 으로 화면을 찍어 리포트에 첨부 (접근)

패턴이 같습니다. **CLI 든 MCP 든 도구는 접근을 제공하고, Skill 이 절차를 정의합니다.** 도구를 바꿔도 Skill 이 유지되고, Skill 을 개선해도 도구 연결은 그대로입니다.

<Callout type="info" title="Skill 없이도 되는 순간">
"이 URL 스크래핑해줘"·"이 이슈 목록 가져와" 처럼 한 번의 호출로 끝나는 작업은 CLI 나 MCP 단독으로 충분합니다. Skill 이 필요한 건 **여러 단계로 이뤄지고 매번 같은 기준으로 결과를 내야 하는** 반복 워크플로우입니다. Eval 처럼 팀 규칙을 담는 작업, 코드 리뷰처럼 형식이 고정되어야 하는 작업이 대표적입니다.
</Callout>

## 핵심 포인트 정리

1. **도구는 가능성, Skill 은 일관성**: CLI · MCP 는 Claude 가 닿을 수 있는 범위를 넓히고, Skill 은 그 도구를 매번 같은 방식으로 쓰게 만듭니다. 하나만 있으면 절반만 해결됩니다
2. **CLI 든 MCP 든 패턴은 동일**: 도구가 접근을 제공하고 Skill 이 절차를 정의합니다. 코드 리뷰든 버그 리포트든, 실전 워크플로우는 전부 이 결합으로 설계됩니다
3. **관심사 분리가 유연성을 만든다**: 도구를 바꿔도 Skill 이 유지되고, Skill 을 개선해도 도구 연결은 그대로입니다. 예를 들어 리뷰 Skill 의 CLI 부분만 MCP 로 바꾸어도 Skill 의 리뷰 기준은 그대로 작동합니다

## FAQ

- **Q: MCP 서버에도 사용 지침이 포함돼 있다고 들었습니다. 그럼 Skill 이 필요 없지 않나요?**
  - A: MCP 서버가 제공하는 설명은 **"이 Tool 을 어떻게 호출하는가"** 에 한정됩니다 (예: "이 쿼리 문법을 써라"). Skill 이 담는 건 **"이 도구를 업무 흐름에서 언제 · 어떤 순서로 · 어떤 형식으로 쓸까"** 입니다. 범위가 다릅니다. MCP 지침은 Tool 수준, Skill 은 워크플로우 수준

- **Q: 하나의 Skill 이 여러 외부 도구를 쓸 수 있나요?**
  - A: 가능합니다. 리서치 Skill 이 gh CLI 로 PR 을 가져오고, Linear MCP 로 관련 이슈를 검색하고, Chrome DevTools MCP 로 운영 대시보드를 확인하는 식입니다. 반대로 같은 외부 도구를 여러 Skill 이 공유할 수도 있습니다

- **Q: 외부 도구만 연결한 상태에서도 잘 되는 것 같은데요?**
  - A: 단발성 작업엔 충분합니다. Skill 이 필요한 순간은 **같은 작업을 여러 번 반복해야 하고, 매번 같은 결과가 나와야 할 때** 입니다. 혼자 쓰는 한두 번짜리 작업엔 과할 수 있지만, 팀이 공유하거나 주기적으로 돌리는 워크플로우에서는 Skill 이 결정적입니다

## Chapter 07 을 마치며

이번 Chapter 에서 Claude 의 **접근 범위** 를 외부로 넓히는 세 가지 경로와, 그 도구를 일관된 워크플로우로 쓰는 법을 배웠습니다.

- **CLI** — 개발자가 이미 쓰던 터미널 도구를 Claude 에게 빌려주기
- **MCP** — CLI 로 못 닿는 영역 (실행 중 앱 · OAuth 서비스 · 팀 공유 설정) 을 표준 프로토콜로 연결
- **Custom MCP** — 공개된 MCP 가 없는 내부 시스템을 직접 감싸기
- **Skill 과의 결합** — 접근 권한에 사용 절차를 더해 반복 워크플로우를 일관되게

하지만 Claude 가 **언제 · 어떤 조건에서 · 무엇을 멈춰야 하는지** 는 아직 통제 아래에 있지 않습니다. AI 에게 맡기는 일이 반복될수록, "이런 상황에서는 꼭 이 검증을 돌려라"·"이 파일을 고친 뒤엔 꼭 포맷팅을 돌려라" 같은 **자동 실행 규칙** 이 필요해집니다.

다음 Chapter 에서는 Claude 의 실행 흐름 자체를 제어하는 **Hooks** 와, 대화 컨텍스트를 분리하는 **Custom Agent** 를 배웁니다.
