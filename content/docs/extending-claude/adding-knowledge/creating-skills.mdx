## Overview

앞 레슨에서 Skill 의 원리를 배웠습니다. 이번 레슨은 직접 Skill 을 하나 만들어 봅니다. Chapter 06 Lesson 02 에서 만든 `/commit` Custom Command 는 사용자가 슬래시로 호출해야만 동작했습니다. 같은 내용을 Skill 로 옮기면 **"변경사항 커밋해줘" 라고 자연어로 말해도 Claude 가 알아서 호출** 하게 됩니다.

두 가지 방법을 모두 체험합니다 — **수동 작성** (SKILL.md 를 직접 타이핑) 과 **Skill Creator 자동 생성** (공식 플러그인에게 맡기기). 같은 결과를 만드는 두 경로를 나란히 본 다음 "언제 어느 쪽을 쓰는가" 를 정리합니다.

### 학습 목표

- `.claude/skills/<이름>/SKILL.md` 구조로 Skill 을 수동 작성할 수 있습니다
- `description` 의 긍정·부정 트리거로 자동 호출 정확도를 높일 수 있습니다
- Skill Creator 로 Skill 을 자동 생성하는 방법과 수동 작성과의 차이를 비교할 수 있습니다

### 시작하기 전 확인사항

- 실습 프로젝트의 시작 브랜치로 전환합니다 (`git checkout ch06-04`)
- Skill Creator 플러그인이 설치되어 있어야 합니다 (`/plugin` → marketplace 에서 `skill-creator` 설치)

`ch06-04` 브랜치에는 앞 레슨에서 만든 `.claude/commands/commit.md` 가 포함되어 있습니다. 이 Command 를 Skill 로 옮기는 것이 이번 실습의 출발점입니다.

## Step 1: Skill 폴더 만들기

Skill 은 마크다운 파일 하나가 아니라 **폴더 단위** 로 저장합니다. 프로젝트 루트에 `.claude/skills/commit/` 폴더를 만듭니다.

```shell
mkdir -p .claude/skills/commit
```

폴더 이름이 곧 Skill 이름이 됩니다. 이 폴더 안에 `SKILL.md` 본체와 선택적으로 참조 파일들이 들어갑니다.

```plain text
.claude/skills/commit/
  SKILL.md                    # 본문 (2단계에서 로드)
  references/                 # 참조 자료 (3단계에서 필요 시)
    commit-conventions.md
    breaking-change-guide.md
```

## Step 2: SKILL.md 직접 작성하기

`.claude/skills/commit/SKILL.md` 를 다음과 같이 작성합니다.

```markdown
---
name: commit
description: Conventional Commit 형식으로 변경사항을 커밋합니다. 코드 변경 후 커밋 요청, "커밋해줘", "변경사항 정리해줘" 요청 시 사용. 단순 git 명령어 질문에는 사용하지 않습니다.
---

# Commit Skill

## 커밋 절차

1. `git status` 와 `git diff --staged` 로 전체 상태를 파악합니다
2. 스테이징된 변경이 없으면 `git diff` 와 `git status` 로 수정된 파일·untracked 파일을 확인하고 관련 파일을 스테이징합니다
3. 변경 내용을 분석해 Conventional Commit 형식으로 메시지를 작성하고 자동으로 커밋합니다

## 커밋 메시지 규칙

- 형식: `<type>(<scope>): <description>`
- type: feat, fix, refactor, test, docs, chore
- scope: 변경된 주요 모듈·컴포넌트 이름
- description: 영어, 소문자, 현재형, 50자 이내

## 예시

- `feat(todo): add filter tabs for completion status`
- `fix(todo-item): resolve checkbox toggle not persisting`
```

Custom Command 와 가장 큰 차이는 **frontmatter 의 `description`** 입니다. 이 한 줄이 Claude 의 자동 호출 판단 기준이 됩니다.

### description 의 세 조각

위 `description` 을 세 부분으로 분해하면 이렇게 됩니다.

- **하는 일**: "Conventional Commit 형식으로 변경사항을 커밋합니다"
- **긍정 트리거**: "코드 변경 후 커밋 요청, '커밋해줘', '변경사항 정리해줘' 요청 시 사용"
- **부정 트리거**: "단순 git 명령어 질문에는 사용하지 않습니다"

사용자가 "git rebase 가 뭐야?" 라고 물으면 Claude 는 이 Skill 을 로드하지 않습니다. "작업 끝났으니 커밋해줘" 라고 자연어로 요청하면 Claude 가 알아서 Skill 을 꺼냅니다.

<Callout type="info" title="좋은 description 작성법">
`[하는 일] + [사용 시점] + [사용하지 않을 때]` 세 조각으로 쓰는 것이 원칙입니다. 처음에는 긍정 트리거만 적고 실행해보다가, 원치 않는 상황에서 자동 호출되면 그때 부정 트리거를 덧붙이는 방식이 현실적입니다.
</Callout>

## Step 3: 자동 호출 확인하기

Skill 이 수동·자동 두 방식 모두로 호출되는지 확인합니다.

**수동 호출**

```plain text
/commit
```

**자동 호출**

```plain text
변경사항 커밋해줘
```

자동 호출이 성공하면 프롬프트 밑에 `Skill(commit) Successfully loaded skill` 이 표시되고, Claude 가 커밋 절차를 바로 시작합니다.

<Lesson04SkillAutoTrigger />

수동·자동이 모두 동작하면 앞서 만든 Custom Command 는 삭제합니다. Skill 이 Command 의 기능을 포함하므로 같은 이름을 두 곳에 둘 이유가 없습니다.

```shell
rm .claude/commands/commit.md
```

## Step 4: Skill Creator 로 같은 Skill 자동 생성하기

수동으로 만들어 봤으니 이번엔 같은 Skill 을 **Skill Creator** 에게 맡겨 봅니다. Skill Creator 는 Anthropic 공식 플러그인으로, 대화를 통해 Skill 을 생성합니다.

비교를 위해 방금 만든 Skill 을 다른 이름으로 옮겨둡니다.

```shell
mv .claude/skills/commit .claude/skills/commit-manual
```

그리고 Skill Creator 를 호출합니다.

```plain text
/skill-creator Conventional Commit 형식으로 커밋하는 Skill 을 만들어줘
```

Skill Creator 는 대화로 요구사항을 정리합니다 — "어떤 타입을 허용할지, 메시지 길이 제한이 있는지, 다국어 지원은 어떻게 할지" 같은 질문을 던지고, 응답을 받아 SKILL.md 를 자동으로 구성합니다.

### 수동 작성과의 두 가지 차이

**description 튜닝**: Skill Creator 는 description 을 다양한 샘플 프롬프트로 돌려보면서 긍정·부정 트리거가 실제로 원하는 대로 동작하는지 확인합니다. 수동으로 쓸 때 감으로 적어야 했던 부분을 자동으로 조정합니다.

**구조 설계**: 워크플로우 단계를 분석해 어떤 내용을 SKILL.md 본문에 두고 어떤 것을 참조 파일로 분리할지 제안합니다. Progressive Disclosure 3단계를 의식한 설계를 자동으로 해줍니다.

생성된 `.claude/skills/commit/SKILL.md` 를 방금 옮겨둔 `.claude/skills/commit-manual/SKILL.md` 와 비교해 보면, 뼈대는 비슷하지만 description 의 구체성과 참조 파일 분리 수준에서 차이가 보입니다.

## Step 5: 수동과 자동, 언제 무엇을 쓰는가

두 방법을 다 겪어봤으니 선택이 단순해집니다.

| | 수동 작성 | Skill Creator |
|---|---|---|
| **속도** | 파일 하나만 쓰면 끝 | 대화 왕복 1~3 번 |
| **초기 품질** | description 은 감으로 | description 자동 튜닝 |
| **구조 제안** | 없음 (직접 판단) | 참조 파일 분리까지 제안 |
| **적합한 상황** | 이미 패턴이 명확한 단순 Skill | 워크플로우가 복잡하거나 description 이 애매한 Skill |

**수동 작성** 이 유리한 경우 — Command 를 Skill 로 옮기는 것처럼 원본 프롬프트가 이미 다듬어져 있고, 할 일이 한두 단계로 명확한 경우입니다.

**Skill Creator** 가 유리한 경우 — 아직 워크플로우가 한 페이지짜리 문서 수준이거나, description 이 원치 않는 상황에서 자주 로드될 것 같은 경우입니다.

## Skill 이 자라는 방향

Skill 은 처음부터 완벽할 필요가 없습니다. 수동으로 한 파일만 썼다가, 사용하면서 참조 자료를 하나씩 덧붙이는 방식이 가장 현실적입니다.

```plain text
.claude/skills/commit/
  SKILL.md                         # 시작: 본문만
  references/
    commit-conventions.md          # 나중 추가: 타입별 예시 모음
    breaking-change-guide.md       # 나중 추가: breaking change 감지 규칙
```

SKILL.md 안에서 "breaking change 가 감지되면 `references/breaking-change-guide.md` 를 읽는다" 처럼 조건부로 참조 파일을 가리키면, Claude 는 **그 조건을 만났을 때만** 참조 파일을 읽습니다. Progressive Disclosure 의 3단계가 실제로 동작하는 모습입니다.

## 핵심 포인트 정리

1. **Skill 은 폴더 단위**: `.claude/skills/<이름>/SKILL.md` 가 본체이고, 참조 파일은 같은 폴더 안에 둡니다. 참조 파일은 필요할 때만 로드됩니다
2. **description 이 자동 호출의 열쇠**: 하는 일 + 긍정 트리거 + 부정 트리거 세 조각으로 쓰면 Claude 의 판단 정확도가 높아집니다
3. **수동과 자동은 용도가 다릅니다**: 이미 명확한 패턴을 옮길 땐 수동이 빠르고, 워크플로우가 아직 정리되지 않았으면 Skill Creator 가 description 튜닝과 구조 제안까지 도와줍니다

## FAQ

- **Q: 같은 이름의 Command 와 Skill 이 둘 다 있으면 어떻게 되나요?**
  - A: 두 곳에 같은 이름이 있으면 Claude 가 어느 쪽을 부를지 혼란스러워집니다. Skill 로 옮긴 뒤엔 `.claude/commands/` 의 원본을 꼭 삭제합니다

- **Q: Skill 이 원치 않는 상황에서 자동 로드됩니다**
  - A: description 에 부정 트리거를 추가합니다. 예를 들어 `"단순 git 명령어 질문에는 사용하지 않음"` 처럼 제외 조건을 명시하면 Claude 가 해당 상황에서 Skill 을 로드하지 않습니다

- **Q: Skill 의 자동 호출을 끄고 싶을 때는요?**
  - A: SKILL.md 의 frontmatter 에 `disable-model-invocation: true` 를 추가하면 사용자가 `/skill-name` 으로 직접 부를 때만 로드됩니다. 배포·데이터 삭제처럼 실수가 치명적인 작업에 권장합니다
