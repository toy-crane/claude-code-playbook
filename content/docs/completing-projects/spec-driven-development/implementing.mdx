## Overview

Lesson 5에서 plan.md를 확정했습니다. 이제 plan의 Task를 실제 코드로 옮길 차례입니다. **`/execute-plan`은 plan을 입력으로 받아 Task를 한 번에 하나씩 TDD로 구현하고, code-reviewer가 5차원으로 리뷰한 뒤, 사용자가 최종 확인하는 흐름을 한 호출로 묶습니다.**

### 학습 목표

- /execute-plan의 4단계 워크플로우를 구별합니다
- 수용 기준이 RED → GREEN 사이클의 입력으로 어떻게 작동하는지 설명합니다
- code-reviewer의 5차원과 Critical/Important/Suggestion 분류 기준을 적용합니다

## Step 1: plan.md를 입력으로 호출한다

```
/execute-plan feedme
```

AI가 plan.md를 읽고 Task 의존성을 분석해 실행 순서를 결정합니다. 사용자는 결과만 확인하면 됩니다.

| 검토 대상 | 무엇을 확인하는가 |
|---|---|
| Task 의존성 | 공유 파일·import 관계·데이터 흐름 |
| Required Skills | plan에 나열된 각 SKILL.md를 미리 로드 — 구현 중 규약이 일관되게 적용 |

순서가 정해지면 그 근거가 learnings.md에 기록되고 다음 단계로 갑니다.

## Step 2: Task 순서대로 TDD로 구현한다

각 Task를 한 번에 하나씩 구현합니다. 핵심은 **수용 기준이 RED → GREEN 사이클의 입력**이 된다는 점입니다.

```
1. 수용 기준 읽기 — "URL 입력 후 변환 버튼 클릭 → 5초 이내 Markdown이 결과 영역에 표시"
2. RED — 그 기준을 검증하는 테스트를 먼저 작성. 테스트는 실패한다.
3. GREEN — 기준을 충족하는 최소 코드 구현. 테스트가 통과한다.
4. bun run build + 영향받은 테스트 실행
5. Task당 conventional commit 1개
6. plan.md에서 Task를 완료로 표시
```

### TDD를 적용하는 부분 / 적용하지 않는 부분

**모든 수용 기준을 테스트로 표현하지는 않습니다.** 코드로 증명 가능한 부분만 TDD로 가고, 디자인 판단·시각 검증은 Human review로 남깁니다.

| 수용 기준 유형 | TDD 적용 |
|---|---|
| 외부 관찰 가능한 동작 (텍스트, 요소 개수, 시각 상태) | ✅ Vitest / Playwright |
| API 응답 형태·라우팅·상태 변화 | ✅ Vitest |
| 디자인 판단 (간격, 색감, 타이포 균형) | ❌ Human review |
| 스크린 리더·접근성 느낌 | ❌ Human review |

### Hook이 코드 스타일을 자동 정리

`PostToolUse` Hook(`lint-fix.sh`)이 파일을 수정할 때마다 자동으로 ESLint를 돌립니다. AI가 깜빡 인덴트나 import 정렬을 빠뜨려도 사용자가 손댈 일이 없습니다.

### 실패 시 우회 금지

빌드나 테스트가 실패하면 우회하지 않고 근본 원인을 찾습니다 -- 빌드 skip, 테스트 disable, 에러 swallow는 기술 부채를 가리는 단기 우회일 뿐입니다.

## Step 3: code-reviewer가 5차원으로 리뷰한다

모든 Task 구현이 끝나면 `code-reviewer` 에이전트가 별도 컨텍스트에서 변경 사항을 5차원으로 평가합니다.

| 차원 | 무엇을 보는가 |
|---|---|
| **정확성** | spec/task가 명시한 동작을 코드가 실제로 수행하는가? 엣지 케이스(null, 빈 값, 경계값)가 처리되는가? |
| **가독성** | 다른 엔지니어가 설명 없이 이해할 수 있는가? 이름이 서술적인가? |
| **아키텍처** | 기존 패턴을 따르는가? 모듈 경계가 유지되는가? 순환 의존이 없는가? |
| **보안** | PII 노출, 권한 우회, 인증 누락 같은 취약점이 없는가? |
| **성능** | 명백한 비효율이 있는가 (불필요한 리렌더, N+1 쿼리, 메모리 누수)? |

리뷰어는 발견 사항을 3분류로 묶어 보고합니다.

| 분류 | 기준 | 처리 |
|---|---|---|
| **Critical** | 머지하면 사용자/시스템에 즉각 손해 (PII 노출, 보안 취약점, 데이터 손실) | 직접 수정 + 영향 테스트 재실행 |
| **Important** | 동작은 하지만 다음 작업 비용을 누적시킴 (유지보수 부담, 중복) | 직접 수정 (spec 범위 밖이면 기각 + 근거 기록) |
| **Suggestion** | 가독성·취향, 동작·유지보수 영향 없음 (변수명, 미세 리팩터링) | Step 4 보고에만 언급, 반영은 선택적 |

판정 결과가 learnings.md에 기록됩니다.

## Step 4: Human Review와 learnings.md 기록

모든 Task와 리뷰가 끝나면 사용자에게 요약이 제시됩니다.

```
- Task별 수용 기준 통과 여부 (pass / partial / fail)
- bun run build + 테스트 결과
- code-reviewer 발견 사항 (Critical / Important / Suggestion 별)
- learnings.md에 기록된 판단 — Task 재정렬·피드백 기각·접근 전환 등
```

사용자가 마지막으로 확인하고 머지를 결정합니다.

### learnings.md — 명백한 인사이트는 즉시 승격

구현 중 명백한 인사이트(예: 같은 에러를 두 번 만났을 때, plan에 빠진 규약을 발견했을 때)는 그 자리에서 Skill·Hook·Rule·CLAUDE.md로 승격됩니다. `applied: rule`로 표시되어, 다음 사이클부터 그 규칙이 자동으로 적용됩니다.

약한 신호(한 번만 보였거나, 패턴인지 불확실한 것)는 `applied: not-yet`으로 두고 누적합니다. 누적된 약한 신호의 처리는 Lesson 8에서 다룹니다.

## 사용자가 끼어들 시점

/execute-plan은 한 번 호출하면 끝까지 진행하지만, 사용자가 결정해야 하는 순간이 네 가지 있습니다.

| 상황 | 사용자 결정 |
|---|---|
| Task가 plan과 다른 순서로 가는 게 합리적 (예: Task 3이 Task 1 출력에 의존) | AI가 재정렬 제안 → 사용자 OK 후 진행. 근거를 learnings에 기록 |
| 진행 중 사용자가 spec 범위 밖 요청을 추가 (예: "비밀번호 재설정도 해줘") | AI가 기각 + 새 feature로 분리 제안. 사용자가 재고 후 결정 |
| 빌드/테스트 실패가 반복되는데 근본 원인이 불명 | AI가 사용자 escalation. 함께 원인 진단 |
| code-reviewer의 Important 항목 처리 | 사용자가 수용/기각 결정 |

판단은 AI가 일차적으로 내리고, **사용자에게 escalate되는 것은 spec 범위·아키텍처 변경처럼 결정이 비싼 경우**입니다.

## 핵심 포인트 정리

1. **수용 기준이 TDD를 이끈다**: 코드로 증명할 수 있는 수용 기준은 RED → GREEN으로 자동화되고, 디자인·접근성 느낌 같은 부분만 Human review로 남습니다
2. **Task 단위로 닫힌다**: 한 Task 구현 → 빌드 → 테스트 → conventional commit → plan 갱신이 한 사이클입니다. Vertical Slice가 끝날 때마다 시스템이 동작 가능 상태에 있습니다
3. **code-reviewer 5차원 + 3분류**: 정확성·가독성·아키텍처·보안·성능을 보고 Critical/Important/Suggestion으로 분류합니다. 처리는 분류에 따라 다르고, 판정은 learnings에 남습니다
4. **사용자 결정점은 비싼 결정에 한정**: 재정렬·기각·escalation 같은 비싼 판단만 사용자에게 올라옵니다 -- 나머지는 AI가 처리합니다

## FAQ

- **Q: 모든 수용 기준을 테스트로 표현해야 하나요?**
  - A: 아닙니다. 코드로 증명 가능한 부분만 TDD로 갑니다. 디자인·시각 판단·스크린 리더 느낌처럼 자동화 어려운 항목은 Human review로 plan에 이미 표기되어 있습니다.

- **Q: code-reviewer의 Suggestion을 무시해도 되나요?**
  - A: 네. Suggestion은 가독성·취향 영역으로, 동작과 유지보수에 영향이 없습니다. 시간이 남으면 반영하고, 그렇지 않으면 보고만 받고 넘어갑니다.

- **Q: AI가 plan과 다른 순서로 진행하면 어떻게 알 수 있나요?**
  - A: AI가 재정렬을 결정하면 사용자에게 알리고 근거를 learnings.md에 남깁니다. 사용자는 OK/거부할 수 있고, 진행 후에도 learnings를 보면 어떤 판단이 들어갔는지 추적할 수 있습니다.
