## Overview

Lesson 3에서 spec.md를 확정했습니다. spec은 "무엇을" 정의하지만, **"어떻게 보이는가"는 정의하지 않습니다.** 같은 spec으로도 입력창을 위에 둘지 옆에 둘지, 결과를 모달에 띄울지 같은 화면에 펼칠지를 정해야 합니다. 이런 레이아웃 결정은 spec에 없습니다.

이 빈칸을 코드 작성 전에 채우는 단계가 **Sketch**입니다. `/sketch-wireframe`은 spec.md를 읽고 HTML 와이어프레임 한 파일을 생성합니다. 텍스트라 AI가 직접 수정하고, 브라우저에서 즉시 확인할 수 있습니다.

### 학습 목표

- /sketch-wireframe의 3단계 워크플로우를 구별합니다
- base 화면과 시나리오 화면의 역할 차이를 설명합니다
- Screen Notes에 무엇을 적고 무엇을 적지 않는지 판단합니다

## Step 1: 기존 화면과 참조 자료 확인

```
/sketch-wireframe feedme
```

AI가 곧장 화면을 그리지 않습니다. 두 가지를 먼저 확인합니다.

| 입력 | 무엇을 읽는가 | feedme.wiki 사이클 |
|---|---|---|
| 코드베이스의 관련 컴포넌트 | 기존 레이아웃 구조: 새 화면이 어디에 얹힐지 | 빈 프로젝트라 영향 없음 |
| `artifacts/feedme/references/` | 참조 이미지가 있으면 구조·정보 계층만 추출 (시각 디자인은 무시) | 없음 |

기존 화면 위에 요소를 추가하는 케이스라면 그 레이아웃을 바탕으로 와이어프레임을 만듭니다. 첫 사이클이라면 spec.md만 가지고 Step 2로 갑니다.

## Step 2: Base Screens, 기본 레이아웃을 먼저 확정한다

AI가 spec.md 시나리오를 화면 단위로 묶고, 첫 화면인 **base 화면**의 HTML 와이어프레임을 생성합니다.

```
3개 화면으로 구성합니다:
1. 기본 화면 (시나리오에 매핑되지 않음, 대표 데이터로 채움)
2. 변환 완료: FEEDME-001, FEEDME-002
3. 변환 실패: FEEDME-003
```

base 화면은 입력창·결과 영역 같은 **레이아웃 골격**을 대표 데이터로 채워둔 상태입니다. 시나리오에 매핑되는 화면은 다음 단계에서 추가합니다.

```
Screen: feedme [col]
├── Header [row]: 로고, 다크모드 토글
├── URL Input [row]: 입력창, 변환 버튼, 지우기 버튼
├── Result [col]: 페이지 제목·저자, 렌더링된 Markdown
└── Actions [row]: 복사, 다운로드, ChatGPT·Claude로 열기
```

AI가 Vite dev server를 백그라운드로 실행하고 브라우저 URL을 알려줍니다.

```
http://localhost:3456/wireframe.html
```

상단 nav에는 화면 전환 버튼과 **Mobile / Desktop 뷰포트 토글**이 함께 들어갑니다. 두 뷰포트 모두에서 레이아웃이 깨지지 않는지 확인합니다.

### 피드백 루프

레이아웃이 마음에 들지 않으면 자연어로 수정을 요청합니다.

```
입력창이 너무 좁다. 변환 버튼을 입력창 안쪽 오른쪽 끝으로 옮겨줘.
```

AI가 wireframe.html을 수정하면 Vite가 변경을 즉시 반영합니다. base 레이아웃이 확정될 때까지 이 루프를 반복합니다.

## Step 3: Scenario Screens, 시나리오마다 화면을 추가한다

base 레이아웃이 확정되면 spec.md의 나머지 시나리오를 화면으로 추가합니다. 모든 시나리오 화면이 같은 wireframe.html에 함께 들어가서, 한 자리에서 비교하며 검토할 수 있습니다.

### 새 화면 vs 기존 화면 매핑

원리는 단순합니다. **레이아웃 구조가 다르면 새 화면, 데이터·상태만 다르면 기존 화면 재사용**입니다.

| 시나리오 차이 | 판정 |
|---|---|
| 빈 결과 영역 → Markdown이 채워진 결과 영역 | 기존 화면 재사용 (대표 데이터로 한 .screen) |
| 메인 화면 → 외부 LLM 열기 모달 | 새 화면 (오버레이 구조 추가) |
| 일반 결과 → 프롬프트 프리셋 + 직접 입력 UI 추가 | 새 화면 (새 요소 배치) |
| Markdown 결과 → 변환 실패 에러 메시지 | 레이아웃 동일하면 기존 화면, 메시지가 결과 영역을 차지하면 새 화면 |

각 시나리오에는 spec의 성공 기준에서 가져온 **구체적 예시 데이터**를 사용합니다. "카드 항목" 같은 자리표시자 대신, 실제 URL과 변환된 Markdown 샘플을 적습니다.

Step 2와 동일한 피드백 루프로 시나리오 화면을 다듬습니다.

## Screen Notes는 화면 흐름만 적는다

각 화면 아래에 **Screen Notes**가 들어갑니다. 화면 진입·인터랙션·전환을 적는 메모입니다. **유효성 규칙과 비즈니스 로직은 spec.md의 영역이므로 여기에는 적지 않습니다.**

좋은 예:

```
트리거: 메인 화면에서 URL 입력 후 변환 버튼 클릭
인터랙션: 결과 영역에 Markdown 표시 → 복사 / 다운로드 / 외부 LLM 버튼
전환: "외부 LLM 열기" 클릭 → 외부 LLM 모달 화면
```

나쁜 예 (비즈니스 로직이 섞임):

```
트리거: 메인 화면에서 URL 입력 후 변환 버튼 클릭
인터랙션: URL이 빈 값이면 에러, 잘못된 URL이면 에러, 변환 실패 시 재시도
```

유효성 규칙(빈 값·잘못된 URL·변환 실패)은 spec.md의 시나리오·성공 기준에 들어갑니다. **Screen Notes에 들어오면 wireframe이 바뀔 때마다 같은 규칙을 다시 봐야 합니다.** spec.md → wireframe 한 방향 흐름을 지키면 한쪽만 고치면 됩니다.

## Wireframe 검토 기준

확정된 wireframe은 `artifacts/feedme/wireframe.html`에 저장됩니다. 사용자가 세 가지만 검토합니다.

1. **모든 spec 시나리오의 UI가 반영되어 있는가?** spec의 시나리오 ID가 어느 .screen의 `data-scenario`에든 등장하는지 확인합니다
2. **컴포넌트 구조가 명확한가?** 입력 영역·결과 영역·액션 영역의 경계가 시각적으로 구분되는지
3. **구체적 예시 데이터가 사용되었는가?** "카드 항목" 같은 자리표시자가 아니라 실제 URL과 Markdown 샘플이 들어가 있는지

## 핵심 포인트 정리

1. **base 화면을 먼저 확정한다**: 레이아웃 골격을 대표 데이터로 채운 첫 화면을 브라우저 피드백 루프로 다듬은 뒤, 시나리오 화면을 그 위에 쌓습니다
2. **모든 시나리오 화면이 한 파일에 모인다**: spec.md의 시나리오마다 화면이 추가되어, 한 자리에서 비교하며 검토할 수 있습니다
3. **레이아웃 구조 vs 데이터·상태 차이**: 구조가 다르면 새 화면, 데이터·상태만 다르면 기존 화면을 재사용합니다
4. **Screen Notes는 화면 흐름만**: 유효성 규칙·비즈니스 로직은 spec.md로 옮깁니다. 두 문서의 경계를 지키면 한쪽만 고쳐도 됩니다

## FAQ

- **Q: 모든 기능에 wireframe이 필요한가요?**
  - A: 아닙니다. API 엔드포인트·비즈니스 로직처럼 UI가 없는 변경은 spec.md만으로 충분합니다. wireframe은 새 화면이나 레이아웃 구조가 크게 바뀌는 feature에서 효과적입니다.

- **Q: Figma 디자인이 있으면 wireframe 단계를 건너뛸 수 있나요?**
  - A: 가능합니다. wireframe은 디자이너 없이 개발자가 직접 레이아웃을 결정해야 할 때의 단계입니다. Figma·Pencil MCP를 연결하면 디자인 도구의 결과를 같은 워크플로우로 받을 수 있습니다.
