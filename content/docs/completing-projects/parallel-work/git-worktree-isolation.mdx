## Overview

Chapter 09에서 SDD 사이클로 한 번에 하나의 기능을 완성하는 흐름을 익혔습니다. 이번 레슨은 그 사이클을 둘 이상 동시에 돌릴 때 가장 먼저 부딪히는 문제, 파일 충돌을 다룹니다. git worktree로 각 Claude에게 독립된 작업 디렉토리를 주고, `claude -w`로 worktree 생성부터 정리까지 한 번에 처리하는 방식을 확인합니다.

### 학습 목표

- git worktree가 일반 브랜치 전환과 어떻게 다른지 설명할 수 있습니다
- `claude -w`로 격리된 에이전트 세션을 만들고 PR까지 마무리할 수 있습니다
- 세션 종료 시 Keep/Remove 선택이 무엇을 남기고 무엇을 지우는지 구별할 수 있습니다

### 시작하기 전 확인사항

- Git 기본 명령어 (commit, branch, push, stash) 사용 가능
- Claude Code 설치 및 인증 완료
- GitHub CLI (`gh`) 설치 및 인증 완료
- VS Code 설치 (Source Control 패널 확인용)
- Git 클린 상태 (`git status` 결과에 커밋되지 않은 변경사항이 없는 상태)

실습 프로젝트의 시작 브랜치로 전환합니다.

```shell
git checkout ch10-02
```

`ch10-02` 브랜치는 이 레슨의 시작점입니다.

## 같은 디렉토리에서는 두 Claude가 동시에 일하지 못한다

![dual-agent-collision](/learn/completing-projects/parallel-work/dual-agent-collision.png)

Claude A에게 마감일 기능을, Claude B에게 우선순위 기능을 동시에 맡긴다고 가정합니다. 두 Claude 모두 `src/components/todo-item.tsx`를 수정해야 합니다. 같은 디렉토리에서 나란히 실행하면 한쪽이 저장한 내용을 다른 쪽이 덮어쓰거나, 원하지 않는 변경이 함께 커밋됩니다.

브랜치를 새로 만들어 전환하는 방법도 완전한 해결은 아닙니다. `feature-a` 브랜치를 만들고 파일을 수정한 채, 다른 기능을 작업하려고 `main`으로 돌아가봅니다.

> ```bash
> git checkout -b feature-a
> # todo-item.tsx 수정, 커밋하지 않음
> git checkout main
> ```

Git은 전환을 거부합니다.

```
error: Your local changes to the following files would be overwritten by checkout:
        src/components/todo-item.tsx
Please commit your changes or stash them before you switch branches.
```

미완성 코드를 커밋하거나 stash로 잠시 치워야 합니다. 두 Claude가 번갈아 쓰는 구조가 되어, 실질적인 병렬 작업이라고 부르기 어렵습니다.

## git worktree: 한 저장소, 여러 작업 디렉토리

<CheckoutVsWorktree />

**git worktree**는 하나의 저장소에서 여러 브랜치를 각각 별도 디렉토리에 동시에 꺼내 쓰는 기능입니다. 각 디렉토리는 파일 상태가 독립적이지만, `.git` 저장소 자체(커밋 히스토리, 원격 설정 등)는 공유합니다.

- 원본 디렉토리: `main` 브랜치
- `worktrees/feature-a/`: `feature-a` 브랜치
- `worktrees/feature-b/`: `feature-b` 브랜치

세 디렉토리가 동시에 존재하므로, Claude A와 Claude B를 서로 다른 디렉토리에서 실행하면 파일이 겹칠 자리 자체가 사라집니다.

### 수동으로 worktree 만들어보기

앞서 만든 `feature-a`의 변경을 되돌리고 worktree로 다시 시도합니다.

> ```bash
> git checkout -- .
> git checkout main
> git branch -D feature-a
> ```

`feature-a`와 `feature-b`를 각각 별도 worktree로 생성합니다.

> ```bash
> git worktree add worktrees/feature-a -b feature-a
> git worktree add worktrees/feature-b -b feature-b
> ```

`-b`는 새 브랜치를 함께 만드는 옵션입니다. 이미 존재하는 브랜치를 꺼내고 싶다면 `-b` 없이 `git worktree add <경로> <브랜치>`로 실행합니다.

이제 `worktrees/feature-a/`를 VS Code로 열고 파일을 수정해봅니다. 원본 디렉토리의 Source Control 패널(Cmd+Shift+G)과 `worktrees/feature-b/`에는 그 변경이 전혀 보이지 않습니다. 에러도 stash도 없이, 두 기능을 동시에 진행할 수 있는 상태가 됩니다.

사용이 끝난 worktree는 목록으로 확인하고 제거합니다.

> ```bash
> git worktree list
> git worktree remove worktrees/feature-a
> git worktree remove worktrees/feature-b
> ```

worktree를 손으로 만들고, 작업하고, 정리하는 흐름을 확인했습니다. 매 작업마다 이 절차를 반복하는 것은 번거롭습니다. `claude -w`는 이 흐름 전체를 한 커맨드로 처리합니다.

## claude -w: worktree 생명주기 자동화

<ClaudeWLifecycle />

`claude -w`는 세 가지를 자동으로 처리합니다. `.claude/worktrees/` 아래에 worktree를 만들고, 그 디렉토리에서 Claude 세션을 시작하고, 세션 종료 시 변경 유무에 따라 자동으로 정리하거나 Keep/Remove를 물어봅니다. `-w`는 `--worktree`의 축약형입니다.

이름을 지정해 실행합니다.

> ```bash
> claude -w deadline
> #         ────────
> #         worktree 이름
> ```

이름을 지정하면 **경로와 브랜치가 예측 가능**해집니다. `.claude/worktrees/deadline/` 디렉토리가 생기고, `worktree-deadline` 브랜치에서 세션이 시작됩니다. 이름을 생략하면(`claude -w`) Claude가 대화 맥락에서 자동으로 이름을 붙입니다.

## [실습] 격리된 에이전트로 기능 하나 PR까지 끝내기

### Step 1. 격리 세션 시작

Todo 앱의 마감일 기능 작업용 격리 세션을 엽니다.

> ```bash
> claude -w deadline
> ```

터미널이 `.claude/worktrees/deadline/` 경로로 이동해 있고, 현재 브랜치가 `worktree-deadline`인지 확인합니다.

### Step 2. 기능 구현 & push

Claude에게 기능을 구현하고 push까지 요청합니다.

> "Todo 목록에 마감일 표시 기능을 추가해줘. 기한이 지난 항목은 빨간색으로 표시. 작업이 끝나면 커밋하고 현재 브랜치 이름(`worktree-deadline`)을 명시해서 push해줘."

<Callout type="info" title="push 시 브랜치 이름을 명시합니다">
worktree에서 새로 만든 브랜치는 기본적으로 `origin/main`을 추적하도록 설정되는 경우가 있습니다. 브랜치 이름을 지정하지 않고 `git push`만 실행하면 main으로 push가 시도될 수 있습니다. main 브랜치가 보호되어 있다면 거부되지만, 보호가 없다면 의도하지 않은 커밋이 main에 올라갑니다. 프롬프트에서 브랜치 이름을 명시해 푸시하도록 요청하거나, 저장소의 main 브랜치 보호 설정을 먼저 확인합니다.
</Callout>

### Step 3. PR 생성 & 병합

push가 완료되면 Claude에게 PR 생성을 지시합니다.

> "PR 만들어줘"

Claude가 `gh pr create`로 PR을 생성합니다. GitHub에서 PR을 리뷰한 뒤 Merge 버튼으로 병합합니다. 이 시점에 원격 `main`에는 마감일 기능이 반영되어 있습니다.

## 세션이 끝나면 무엇이 남는가

![worktree-exit-prompt](/learn/completing-projects/parallel-work/worktree-exit-prompt.png)

Claude 세션을 종료하면(Ctrl+C 또는 `/exit`), worktree 상태에 따라 동작이 달라집니다.

- **변경이 전혀 없는 경우**: 프롬프트 없이 worktree 디렉토리와 브랜치가 자동으로 삭제됩니다
- **커밋이나 수정이 남아 있는 경우**: Keep/Remove 선택 프롬프트가 뜹니다
  - **Keep worktree**: `.claude/worktrees/<name>/`에 디렉토리와 브랜치가 그대로 남습니다
  - **Remove worktree**: 로컬의 변경과 커밋이 전부 삭제됩니다

push나 PR 병합까지 끝낸 작업은 Remove를 선택해도 원격에 이미 반영되어 있으므로 안전합니다. 반대로 push하지 않은 상태에서 Remove를 선택하면, 그 작업은 복구하기 어렵습니다.

### 중단된 작업 이어가기

Keep을 선택해 worktree를 남겼다면, 나중에 같은 worktree에서 세션을 이어갈 수 있습니다.

> ```bash
> claude -w deadline --resume
> #                  ────────
> #                  이전 세션 목록에서 선택
> ```

`--resume` 대신 `--continue`를 쓰면 가장 최근 세션을 바로 이어서 시작합니다.

## 핵심 포인트 정리

1. **git worktree는 `.git`을 공유하면서 파일 상태만 분리합니다**: 하나의 저장소에서 여러 브랜치를 각각 다른 디렉토리에 꺼내 둘 수 있으므로, 두 Claude가 같은 파일을 동시에 작업해도 서로의 변경이 겹치지 않습니다
2. **`claude -w <name>`은 worktree의 생성·사용·정리를 한 커맨드로 묶어줍니다**: 이름을 지정하면 경로와 브랜치가 예측 가능하고, 세션을 끝낼 때 변경 유무에 따라 자동 정리 또는 Keep/Remove 선택이 주어집니다
3. **Keep은 "이어서 할 수 있게 남긴다"는 뜻입니다**: Keep한 worktree는 `claude -w <name> --resume`으로 이어갈 수 있고, push나 PR까지 끝난 작업은 Remove를 선택해도 안전합니다

## FAQ

- **Q: git worktree를 만들면 디스크 공간이 두 배가 되나요?**
  - A: `.git` 디렉토리는 공유하므로 저장소 히스토리가 복제되지는 않습니다. 파일 복사본만 추가되어 node_modules를 제외하면 수십 MB 수준입니다. 다만 worktree마다 `bun install`이 필요하므로 node_modules 크기가 누적될 수 있습니다. 사용이 끝난 worktree는 `git worktree remove`로 바로 정리하는 편이 좋습니다

- **Q: push 없이 세션을 종료하면 어떻게 되나요?**
  - A: 커밋이나 변경이 있으면 Keep/Remove 프롬프트가 뜹니다. Keep을 선택하면 `claude -w <name> --resume`으로 이어갈 수 있고, Remove를 선택하면 로컬 변경이 사라집니다. 중요한 작업은 반드시 push하거나 PR로 올린 뒤에 종료합니다

- **Q: 이름을 지정하지 않고 `claude -w`만 실행하면 어떤 이름이 생성되나요?**
  - A: Claude가 대화 맥락에서 적절한 이름을 자동으로 붙입니다. 명확한 주제가 있을 때는 이름을 직접 지정하는 편이 `--resume`으로 이어갈 때 찾기 쉽습니다

## 이어서 배울 내용

각 Claude에게 독립된 작업 디렉토리를 주는 방법을 익혔습니다. 다만 worktree로 공간만 나눠 둔 상태에서는, 여러 Agent가 서로의 진행 상황을 모른 채 같은 API 형식을 제각기 가정해 충돌할 수 있습니다. 다음 레슨은 여러 Agent가 공유 Task 리스트와 메시징으로 협업하는 Agent Teams를 다룹니다.
