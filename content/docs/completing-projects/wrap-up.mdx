## SDD: 도구를 하나의 사이클로 묶다

Part 2 에서 익힌 Rules·Skills·MCP·Hooks·Custom Agent 가 흩어진 도구가 아니라 하나의 정해진 사이클의 부품으로 들어옵니다. SDD 는 이 도구들을 6단계의 정해진 자리에 배치해 한 번의 위임으로 사이클을 닫게 만드는 프레임워크입니다.

### 6단계와 Spec 의 무게

<SddCycle />

- **6단계 흐름**: Ideate → Specify → Sketch → Plan → Build → Compound. 입력은 한 줄 아이디어지만, 매 단계의 산출물(spec.md, wireframe.html, plan.md, code, learnings.md)이 다음 단계의 입력으로 직접 이어집니다.
- **Spec 이 사이클 전체를 결정합니다**: 같은 Plan, 같은 Build 도구를 써도 Spec 이 모호하면 결과가 흔들립니다. 변경 비용이 높은 결정(가격 모델, 데이터 스키마, 인증 방식)은 Spec 단계에서 먼저 드러내고, 비용이 낮은 결정(색상, 마이크로카피)은 기본값에 맡깁니다.
- **WHAT 과 HOW 의 경계**: spec.md 는 "사용자가 무엇을 보게 되는가" 만 담고, "어떻게 구현하는가" 는 plan.md 로 미룹니다. 두 단계를 섞으면 양쪽의 검토가 모두 약해집니다.

### 단계마다 다른 검증 장치

Spec 만 정확하다고 사이클이 닫히지는 않습니다. 각 단계가 자기 산출물을 검증하는 도구를 따로 가집니다.

- **수용 기준이 자동 테스트로**: spec 의 수용 기준이 RED → GREEN 테스트의 입력으로 변환되어 Build 사이클을 이끕니다. 관찰 가능한 동작은 모두 자동 검증으로 가고, 디자인·접근성처럼 인간 판단이 필요한 부분만 Human review 로 남깁니다.
- **plan-reviewer 의 4축 정합성**: Plan 단계가 끝나면 plan-reviewer 가 spec ↔ wireframe ↔ plan 의 정합성을 4축으로 확인합니다. spec 에 없는 기능이 plan 에 들어가거나, wireframe 의 상태가 plan 의 Task 로 분해되지 않은 경우가 여기서 잡힙니다.
- **code-reviewer 의 5차원 머지 전 점검**: Build 가 끝난 코드를 정확성·가독성·아키텍처·보안·성능 다섯 축으로 검토합니다. Critical 오류가 나오면 그 오류 자체를 fix 하기 전에 spec 의 성공 기준이 부족하지 않았는지 먼저 살핍니다.

### 약한 신호의 누적과 Compound

같은 사이클을 여러 번 돌리면 두 종류의 학습이 쌓입니다.

- **강한 신호는 즉시 승격**: 같은 에러가 두 번 이상 반복되거나 같은 패턴의 수정이 매번 손으로 들어가면, /execute-plan 이 끝난 직후 Rule 이나 Hook 으로 올라옵니다. 다음 사이클부터는 자동으로 적용됩니다.
- **약한 신호는 누적된 뒤 Compound**: 한 사이클 안에서는 우연 같지만 3~5 사이클이 쌓이면 패턴으로 보이는 신호가 있습니다. learnings.md 에 모아 두었다가 /compound 단계에서 회고로 꺼내고, Skill 이나 CLAUDE.md 의 한 줄로 승격합니다.
- **도구 갱신은 다음 사이클을 더 빠르게**: 첫 프로젝트의 learnings 가 두 번째 프로젝트의 도구로 들어가면, 같은 종류의 작업이 처음의 절반 시간에 닫힙니다. 사이클을 도는 비용 자체가 사이클을 거듭할수록 줄어듭니다.

## 병렬 작업: Context 격리에서 협업으로

한 명의 Claude 가 한 번에 한 가지 일을 하는 것으로 충분하지 않은 순간이 옵니다. 두 갈래의 가설을 동시에 시도하거나, 서로 다른 레이어를 동시에 손대야 할 때, 격리와 협업은 다른 도구가 필요합니다.

### Git Worktree, 같은 .git 위 별도 작업 공간

- **격리는 디렉토리 단위로**: 같은 디렉토리에서 두 Claude 가 동시에 일하면 파일 충돌이 즉시 발생합니다. git worktree 는 한 저장소(.git)를 공유하면서 작업 디렉토리만 분리해, 두 Claude 가 같은 파일을 동시에 수정해도 서로 영향을 주지 않습니다.
- **`claude -w <name>` 한 명령으로 생명주기 자동화**: 워크트리 생성, 진입, 종료 후 정리(Keep / Remove)까지 한 명령에 묶입니다. 중단된 작업은 워크트리만 남겨 두면 다음 세션에서 그대로 이어갈 수 있습니다.

### Subagent 와 Agent Teams 의 분기

<SubagentVsTeams />

- **Subagent 는 결론이 짧은 위임을 위한 도구**: 입력은 방대하지만 결론이 한 페이지로 정리되는 작업(대규모 탐색, 테스트 로그 분석)을 별도 컨텍스트에서 처리하고, 메인 대화에는 요약만 돌아옵니다. 메인 컨텍스트의 오염을 막는 것이 1차 목적입니다.
- **Agent Teams 는 팀원 간 직접 대화가 필요할 때**: 여러 팀원이 독립 컨텍스트에서 병렬로 돌면서 공유 Task 리스트와 peer-to-peer 메시지로 협업합니다. Subagent 와 달리 구현 도중에 팀원끼리 직접 대화할 수 있습니다.
- **분기 기준은 한 가지**: "팀원끼리 구현 중 대화해야 하는가". 다각도 병렬 리뷰, 경합 가설 디버깅, 크로스레이어 구현이 그 대화가 필요한 대표 상황입니다. 그 외에는 Subagent 로 충분합니다.

## 개인 프로젝트: 사이클을 내 손으로 닫기

학습의 마지막은 배운 도구로 한 사이클을 처음부터 끝까지 독립으로 돌리는 것입니다. 강사가 만든 예제를 따라 치는 것과 본인 프로젝트로 6단계를 통과하는 것은 같은 동작이 아닙니다.

### 목표는 결과물이 아니라 사이클 완주

- **3시간 안에 닫을 수 있는 크기**: 페이지 2~3 개, 핵심 기능 1~2 개, 시나리오 5~7 개. 멋진 프로젝트가 아니라 6단계를 빠짐없이 한 번 돌리는 경험이 자산입니다.
- **건너뛰지 않기**: Spec 없이 바로 Plan, Plan 없이 바로 Build 처럼 단계를 생략하면 사이클이 닫히지 않습니다. 작은 기능이라도 6단계를 모두 통과시킵니다.
- **단일 Agent 로 충분**: 첫 사이클에서는 Agent Teams 까지 가지 않습니다. 도구를 더 얹기 전에 한 명의 Claude 로 사이클이 닫히는 감각을 먼저 익힙니다.

### 막힐 때는 코드가 아니라 이전 단계로

- **막힘은 직전 단계 산출물의 신호**: Plan 이 막히면 Spec 이 모호하다는 뜻이고, Build 가 어긋나면 Plan 의 분해가 거칠다는 뜻입니다. 코드를 손으로 고치는 대신 직전 산출물(spec.md, plan.md)을 다시 봅니다.
- **문서 수정 비용이 항상 더 낮습니다**: 코드는 수정하면 테스트, 빌드, 리뷰가 다시 돌아야 하지만, 문서는 한 번 고치면 다음 단계가 다시 정확하게 흐릅니다. 사이클 안에서 가장 싼 수정 지점부터 찾는 것이 SDD 의 비용 구조입니다.
