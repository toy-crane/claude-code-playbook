## Overview

앞 레슨에서 CLAUDE.md를 **내가 직접 써서 건네는 업무 매뉴얼**로 그렸습니다. 매뉴얼에는 팀 합의만 남기고 개인 습관은 뺐습니다. 그런데 의문이 남습니다 — "비유는 빼고 정의만 간결하게 써줘" 같은 **나만의 교정**은 대체 어디에 쌓여야 하나요?

이번 레슨은 그 답으로 **Memory**를 소개합니다. Memory는 **내가 쓰는 매뉴얼**이 아니라 **Claude가 대화를 관찰하면서 스스로 쓰는 노트**입니다. 세션이 끝나도 남고, 다음 세션이 시작될 때 Context에 자동으로 다시 실립니다.

### 학습 목표

- Memory가 세션 간 학습을 유지하는 원리를 **설명합니다**
- CLAUDE.md와 Memory의 역할 차이를 **구분합니다**
- Memory에 담아야 할 것과 담으면 안 되는 것을 **판단합니다**

## "그렇게 말고"를 매 세션 반복하고 있다면

같은 교정을 매 아침 다시 하고 있다면 신호입니다.

- "비유는 빼고 정의만 간결하게 써줘"
- "나는 React는 익숙하고 CSS는 약해. 배경 설명 깔아줘"

세션을 새로 열면 Claude는 다시 비유를 가득 넣은 설명을 돌려주고, 리뷰는 평범한 체크리스트로 돌아갑니다. 앞 레슨에서 본 **Stateless**가 여기서도 그대로 동작합니다 — 어제 해준 교정은 오늘 아침에 전부 사라져 있습니다.

이걸 해결하겠다고 CLAUDE.md에 쓸 수도 있지만 바로 막힙니다. `./CLAUDE.md` 는 git으로 **팀 전체**에 공유되고, `~/.claude/CLAUDE.md` 는 **모든 프로젝트**에 적용됩니다. 두 곳 모두 **이 프로젝트에서 내가 자주 교정하는 말**이 들어가기엔 범위가 너무 넓습니다.

이 틈을 메우는 것이 Memory입니다.

## Memory: Claude가 스스로 쓰는 관찰 노트

<ClaudeMdVsMemory />

CLAUDE.md와 Memory의 가장 큰 차이는 **글 쓰는 사람이 다르다**는 것입니다.

- **CLAUDE.md** — 내가 직접 써서 건네는 매뉴얼. 작성 주체는 **나**
- **Memory** — Claude가 대화를 관찰하면서 스스로 적어두는 노트. 작성 주체는 **Claude**

내가 "비유 빼줘"라고 교정하는 순간, Claude는 이 교정을 **다음 세션에도 기억할 가치가 있는 선호**로 보고 자기 노트에 한 줄을 적어둡니다. 내가 `CLAUDE.md` 를 직접 열어 수정하지 않아도, 노트는 대화하는 사이 자연스럽게 쌓입니다.

### 매뉴얼(CLAUDE.md)과 관찰 노트(Memory)의 분담

| | CLAUDE.md | Memory |
|---|---|---|
| **누가 쓰는가** | 내가 직접 | Claude가 자동으로 |
| **무엇을 담는가** | 팀 합의·아키텍처 결정·제약사항 | 개인 선호·교정·작업 패턴 |
| **공유 범위** | git으로 팀 전체 | 로컬, 개인 전용 |
| **충돌 시 우선순위** | 높음 | 낮음 |

한 줄로 정리하면 — **CLAUDE.md는 팀이 합의한 공식 문서, Memory는 Claude가 개인적으로 쌓는 관찰 기록**입니다. 충돌하면 CLAUDE.md가 이깁니다. Memory는 CLAUDE.md가 다루지 않는 빈칸, 즉 **나만의 맥락**을 채우는 자리입니다.

## 저장은 프로젝트별로, 로드는 세션 시작마다

<MemoryLifecycle />

관찰 노트는 프로젝트 단위로 분리되어 `~/.claude/projects/` 아래에 저장됩니다. 프로젝트 경로를 기반으로 폴더가 생기고, 그 안에 `MEMORY.md` 인덱스와 주제별 파일이 들어갑니다.

```
~/.claude/projects/
  -Users-username-my-project/    # 프로젝트 경로 기반
    memory/
      MEMORY.md                   # 인덱스 (앞쪽 200 줄 자동 로드)
      user_preferences.md         # 주제별 파일
      feedback_testing.md
```

`MEMORY.md` 는 매 세션 시작 시 Context에 자동으로 실립니다. 단, 앞쪽 200줄(또는 25KB) 까지만 로드되고 나머지는 필요할 때 주제별 파일에서 가져옵니다. 그래서 Claude는 아무거나 저장하지 않고 **다음 세션에도 유용할 만한 것**만 골라 적도록 동작합니다 — 인덱스를 짧게 유지할 책임은 유저가 아니라 Claude 쪽에 있습니다.

프로젝트별 분리 덕분에 **프로젝트 A의 개인 교정**이 **프로젝트 B 대화**에 끼어들지 않습니다. 또한 Memory는 **로컬 전용** 이라 git에 올라가지 않고 팀원과 공유되지 않습니다. CLAUDE.md가 공유된 매뉴얼이라면, Memory는 내 서랍 안의 수첩입니다.

### 세 가지 시점에 동작

- **세션 시작** — Claude가 저장된 Memory를 읽어 Context에 불러옵니다. 터미널 상단에 메모리가 로드됐다는 표시가 짧게 나타납니다
- **대화 중 자동 감지** — 교정이나 선호 패턴을 감지하면 Claude가 스스로 노트에 한 줄을 추가합니다. 내가 별도 명령을 내리지 않아도 됩니다
- **명시적 저장·삭제 요청** — "이것을 기억해"로 즉시 저장, 반대로 "이 선호는 지워줘"처럼 삭제를 요청하면 해당 항목을 빼 줍니다

## [실습] 개인 선호 하나 기억시키기

Memory가 실제로 동작하는지 직접 확인합니다.

### Step 1: `/memory` 로 현재 상태 확인

Claude Code에서 `/memory` 를 입력합니다.

![memory-command](/learn/starting-conversations/context-management/memory-command.png)

이번 세션에 로드된 메모리 파일들(사용자 전역 `~/.claude/CLAUDE.md`, 프로젝트 `./CLAUDE.md` 등)과 함께, 맨 아래쪽에 **Open auto-memory folder** 항목이 나옵니다. 위쪽 항목들을 고르면 해당 CLAUDE.md가 에디터에서 열리고, 맨 아래 항목을 고르면 자동 Memory 폴더가 파일 탐색기에서 열립니다.

**이 레슨에서 다루는 자동 Memory는 맨 아래 "Open auto-memory folder"입니다.**

### Step 2: 선호 저장 요청

Claude Code에 다음과 같이 입력합니다.

```
설명할 때 비유는 빼고 정의만 간결하게 써줘. 기억해
```

Claude가 이 선호를 Memory에 저장했다는 응답을 확인합니다.

### Step 3: 새 세션에서 검증

`/clear` 로 세션을 초기화하거나 새 터미널에서 `claude` 를 실행합니다. 세션 시작 시 터미널 상단에 메모리 로드 표시가 나타나는지 먼저 봅니다.

그리고 아무 개념이나 설명을 요청합니다.

```
Context Window 가 뭐야?
```

Claude가 비유 없이 정의만 내놓으면 Memory가 정상 동작하는 것입니다. Claude는 기본적으로 비유를 활용해 설명하므로, 비유가 빠져 있으면 관찰 노트가 작용한 결과로 볼 수 있습니다.

## Memory에 담으면 안 되는 것

관찰 노트에 아무거나 쌓이지 않도록 세 가지는 막아둡니다.

- **민감 정보** — API 키·비밀번호·토큰. Memory는 평문으로 저장되므로 민감 정보는 환경 변수나 비밀 관리자로 빼야 합니다
- **팀 규칙** — 팀 전체가 따라야 하는 규칙은 CLAUDE.md에 넣어야 다른 팀원에게도 적용됩니다. Memory는 로컬이라 내 세션에만 작동합니다
- **코드에서 찾을 수 있는 정보** — 기술 스택·디렉토리 구조·명령어. 앞 레슨의 CLAUDE.md 원칙이 그대로 이어집니다. 모델이 스스로 찾을 수 있는 정보를 Memory에 넣어두면 Context만 소모됩니다

## 핵심 포인트 정리

1. **작성 주체의 비대칭**: CLAUDE.md는 **내가** 팀을 위해 써두는 매뉴얼, Memory는 **Claude가** 나를 관찰하면서 적어두는 노트입니다. 둘은 겹치지 않고 보완합니다
2. **개인 선호는 Memory의 자리**: 팀 규칙은 CLAUDE.md, 개인 교정·말투·배경은 Memory. Memory는 로컬 전용이라 팀의 Context를 낭비하지 않습니다
3. **충돌 시 CLAUDE.md가 이깁니다**: Memory는 CLAUDE.md가 다루지 않는 빈칸을 채우는 자리이지, 팀 규칙을 덮어쓰는 자리가 아닙니다

## FAQ

- **Q: Memory가 너무 많아지면 어떻게 되나요?**
  - A: `MEMORY.md` 인덱스는 앞쪽 200줄(또는 25KB) 까지만 자동 로드되므로 뒤쪽이 잘립니다. `/memory` 로 확인하고 쓰지 않는 항목은 Claude에게 삭제를 요청해 정리합니다

- **Q: 팀원과 Memory를 공유할 수 있나요?**
  - A: Memory는 로컬 전용이라 직접 공유되지 않습니다. 팀 전체가 따라야 하는 규칙이라면 CLAUDE.md로 옮기는 것이 맞습니다

- **Q: Memory 파일을 직접 편집해도 되나요?**
  - A: 됩니다. `~/.claude/projects/` 아래 파일을 직접 수정하거나 삭제할 수 있습니다. 다만 **"기억해" / "잊어"**를 쓰는 편이 더 간단합니다

## 이어서 배울 내용

CLAUDE.md와 Memory로 **무엇을 세션 간에 남길지**는 해결했습니다. 남은 질문은 **하나의 대화를 얼마나 길게 이어가야 하는가**입니다. Chapter 03 첫 레슨의 Smart Zone·Dumb Zone으로 돌아가는 질문 — 대화가 길어질수록 품질이 떨어지므로, 가장 효과적인 Context 관리는 **적절한 시점에 끊는 것**입니다.

- 새 대화를 시작해야 하는 시점과 신호
- `/clear` 와 `/compact` 의 차이
