## Overview

Chapter 03 에서 Context Window 가 제한 자원이라는 점, 그리고 CLAUDE.md · Memory 로 세션 간 학습을 남기는 법까지 봤습니다. 남은 질문은 하나입니다 — *하나의 대화를 얼마나 길게 이어갈까.*

Task Sizing 의 답은 몇 년 사이에 크게 바뀌었습니다. 과거에는 *"Context 가 길어지면 무조건 품질이 떨어지니 자주 끊어라"* 가 정석이었지만, 최근 모델들은 같은 주제로 쌓인 긴 대화를 비교적 잘 버팁니다. 그럼에도 Task Sizing 이 여전히 필요한 이유가 있고, 이번 레슨은 그 이유와 실전 기준을 정리합니다.

### 학습 목표

- 예전의 Task Sizing 과 지금의 Task Sizing 이 어떻게 다른지 **구별합니다**
- 새 대화를 시작해야 하는 3 가지 신호를 **판단합니다**
- `/clear` 와 `/compact` 중 언제 무엇을 쓸지 **판단합니다**

## 예전의 Task Sizing, 지금의 Task Sizing

**초기.** Lost-in-the-Middle 이 지금보다 훨씬 심했습니다. 수천 줄짜리 대화가 쌓이면 중반부에 적어둔 지시가 통째로 날아가곤 했고, 안전한 권고는 단순했습니다 — **커밋마다 새 대화.** 하나의 대화에서 기능 A 를 만들고 이어서 기능 B 를 디버깅하면 B 를 다룰 때쯤엔 A 의 도구 응답이 주의력을 대부분 가져가 버렸습니다.

**현재.** 같은 기능 안에서 여러 파일을 오가는 작업은 긴 대화 안에서도 비교적 잘 이어집니다. *"오전에 로그인 API 를 붙이고, 같은 대화에서 에러 메시지를 손보고, 테스트까지 쓴다"* 같은 흐름은 이제 한 잔에 충분히 담깁니다. 예전 규칙을 그대로 적용해 매 커밋마다 `/clear` 를 하면 같은 오리엔테이션을 반복하느라 오히려 비효율이 됩니다.

그럼에도 Task Sizing 이 사라지지 않은 이유는, 긴 Context 가 문제를 일으키는 **방식이 바뀌었기** 때문입니다. 이전에는 *양* 이 문제였다면, 지금은 *성격* 이 문제입니다.

### 지금도 망가지는 순간 — 도메인이 바뀔 때

예를 들어 프론트엔드 로그인 UI 를 완성한 뒤, 같은 대화에서 이어서 백엔드 세션 토큰 만료 로직을 고쳐 달라고 해 봅니다. 모델은 잘 따라오는 듯 답하지만, 결과물을 보면 프론트의 폼 검증 규칙이 백엔드 만료 정책에 슬그머니 새어 들어 있거나, 앞서 Read 한 UI 파일의 네이밍이 백엔드 코드에 끌려 들어옵니다.

모델이 중간 정보를 *놓치는* 게 아니라, 한 컵에 남아 있던 **다른 음료의 향이 새 음료에 섞이는** 상황입니다. 같은 영역에서 많이 따르는 것은 괜찮지만, 성격이 다른 작업을 같은 컵에 부으면 맛이 엉킵니다. 모델이 좋아진 지금 Task Sizing 이 막으려는 것은 이 *오염* 입니다.

## Task Sizing 다시 보기 — 양이 아니라 응집성

![lesson-04-task-sizing-cups](./attachments/lesson-04-task-sizing-cups.png)

컵 은유는 그대로 두되 기준을 바꿉니다. 핵심은 *"얼마나 담았나"* 가 아니라 *"무엇을 섞었나"* 입니다.

- **같은 기능 · 같은 파일 계열** 을 반복해서 수정하는 흐름 → 한 잔에 담아도 괜찮습니다
- **작업의 성격 자체가 달라지는 순간** → 새 잔을 꺼내야 합니다

**수정 → 테스트 → 커밋** 한 사이클을 한 잔으로 잡던 예전 조언은 여전히 안전한 기본값입니다. 다만 이제는 *강제 단위* 가 아니라 *참고 경계* 로 씁니다. 커밋이 끝나도 같은 기능을 이어간다면 한 잔을 더 써도 좋고, 한 커밋 안에서도 성격이 바뀌는 순간이 있다면 중간에 끊어야 합니다.

> 메시지를 보내기 전마다 스스로 물어보세요 — *"이게 지금 컵에 담긴 것과 같은 종류의 작업인가?"*

## 새 잔으로 옮겨야 하는 3 가지 신호

예전의 "무조건 자주" 쪽으로 치우친 신호 목록 대신, 지금 남겨 둘 만한 것은 세 가지입니다.

- **작업의 성격이 바뀔 때 (가장 강한 신호)** — 프론트 → 백엔드, 기능 구현 → 성능 튜닝, 코딩 → 문서 작성처럼 다루는 대상 · 도구 · 판단 기준이 달라지는 순간. 이전 작업의 Context 가 새 작업을 오염시킬 확률이 가장 높은 지점입니다
- **같은 수정 요청이 두 번 빗나갔을 때** — Dumb Zone 때문이라기보다 *실패의 흐름* 이 쌓인 상태입니다. 같은 프롬프트를 새 대화에 그대로 붙여 넣는 것만으로도 결과가 달라지는 경우가 많습니다
- **커밋이 끝났고, 다음 작업이 다른 영역일 때** — 커밋 자체는 자연스러운 경계선이지만, 같은 기능 영역을 이어간다면 반드시 끊을 필요는 없습니다. *커밋 + 도메인 전환* 이 겹치는 지점이 진짜로 끊는 자리입니다

## /clear 와 /compact — 두 가지 리셋 방법

새 터미널을 여는 대신 현재 세션 안에서 컵을 비우는 두 명령어가 있습니다.

### /clear: 크기와 흐름을 동시에 리셋

`/clear` 는 Context 를 완전히 비웁니다. 대화 기록 · 도구 응답 · 누적된 정보가 모두 사라지고, CLAUDE.md 와 Memory 는 세션 시작 시점처럼 다시 로드됩니다. 사실상 새 대화를 여는 것과 같습니다.

`/clear` 의 진짜 힘은 **대화의 흐름까지 끊긴다는 점** 입니다. 드라마에서 주인공이 계속 실패하는 장면이 이어지면 다음 장면도 실패할 것 같은 분위기가 생기듯, 수정 요청이 빗나가는 대화가 쌓이면 Claude 도 *"이 대화에서는 잘 안 되는 중"* 이라는 흐름에 맞춰 답을 내놓습니다. `/clear` 는 이 흐름을 끊고 깨끗한 상태에서 다시 시작하게 합니다.

<Callout type="idea" title="두 번 고쳐도 안 고쳐지면 `/clear`">
세 번째 시도를 실패 흐름 위에서 밀어붙이는 것보다, 같은 프롬프트를 새 대화에 그대로 붙여 넣는 편이 성공 확률이 높습니다. 모델의 문제가 아니라 대화의 궤적이 문제일 때가 많습니다.
</Callout>

### /compact: 요약으로 Context 를 압축

`/compact` 는 대화 기록을 요약해 토큰 수를 줄입니다. 대화의 골자는 남으므로 *같은 작업을 계속 이어가야 하는데 Context 가 꽉 찰 때* 에 쓰입니다. 단, 요약 과정에서 *"A 를 시도했다가 충돌 나서 B 로 바꿨다"* 같은 실패 맥락이 사라져 같은 실수를 반복할 위험이 남습니다.

| 명령어 | 동작 | CLAUDE.md · Memory | 잃는 것 | 추천 상황 |
|---|---|---|---|---|
| `/clear` | 완전 초기화 | 유지 | 전체 맥락 | 작업 성격 전환 · 실패 흐름 리셋 |
| `/compact` | 요약으로 압축 | 유지 | 뉘앙스 | 끊기 어려운 긴 작업을 이어가야 할 때 |

기본 선택은 `/clear` 입니다. `/compact` 는 *끊을 수 없는 긴 작업 하나를 억지로 이어가야 할 때* 에만 꺼내는 예외 도구입니다.

## 핵심 포인트 정리

1. **기준이 양에서 응집성으로 이동**: 모델이 좋아지면서 "자주 끊어라" 는 더 이상 정답이 아닙니다. 같은 기능 · 같은 파일 계열은 한 잔에 담아도 되지만, 작업의 성격이 바뀌는 순간에는 새 잔을 꺼냅니다
2. **가장 강한 신호는 작업 전환**: 프론트 → 백엔드, 구현 → 튜닝처럼 다루는 대상이 달라질 때 이전 Context 가 새 작업을 오염시키기 가장 쉽습니다. 커밋 완료는 이 전환과 겹칠 때만 끊는 신호가 됩니다
3. **기본은 `/clear`, 예외는 `/compact`**: `/clear` 는 Context 크기뿐 아니라 실패의 흐름까지 리셋합니다. `/compact` 는 뉘앙스를 잃으므로, 끊을 수 없는 긴 작업을 어쩔 수 없이 이어가야 할 때만 씁니다

## FAQ

- **Q: 예전에 배운 "커밋마다 새 대화" 규칙은 이제 틀린 건가요?**
  - A: 틀렸다기보다 *과보정* 에 가깝습니다. 모델이 좋아진 지금은 같은 기능 안에서의 여러 커밋을 한 대화로 이어도 문제가 없습니다. 다만 *커밋 완료 + 도메인 전환* 이 겹치는 순간에는 여전히 그 규칙이 유효합니다

- **Q: 그럼 한 대화를 어디까지 이어갈 수 있나요?**
  - A: 정해진 토큰 수가 아니라 *응집성* 이 기준입니다. 같은 기능 · 같은 파일 계열에 머무는 한, 대화가 길어져도 품질이 크게 떨어지지 않는 것이 최신 모델의 경향입니다. 단, 비용과 레이턴시는 길이에 비례해 늘어나므로 실용적으로는 몇 시간 단위로 한 번 끊는 감각이 편합니다

- **Q: `/compact` 만 계속 쓰면 안 되나요?**
  - A: 요약 과정에서 *"A 를 시도했다가 충돌 나서 B 로 바꿨다"* 같은 실패 맥락이 사라집니다. Claude 가 같은 실수를 다시 저지르기 쉬워지므로, 깨끗한 시작이 부분 요약보다 대체로 결과가 좋습니다
