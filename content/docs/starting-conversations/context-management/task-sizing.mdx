## Overview

앞 레슨까지 CLAUDE.md 로 팀 규칙을, Memory 로 개인 선호를 세션 간에 남기는 법을 세웠습니다. 남은 질문은 하나입니다 — *하나의 대화를 얼마나 길게 이어갈까.* Chapter 03 첫 레슨에서 본 대로 대화가 길어질수록 Dumb Zone 이 넓어지고 지침의 저주가 강해집니다. 이번 레슨은 *대화 자체를 짧게 유지하는 기술* 인 **Task Sizing** 을 다룹니다.

Task Sizing 은 프롬프트를 다듬거나 MCP 를 정리하는 다른 어떤 기법보다도 효과가 큽니다. 스펙을 10% 다듬어도 대화가 10 배 길어지면 의미가 없지만, 대화만 제때 끊어도 평범한 프롬프트로도 Smart Zone 에서 시작할 수 있기 때문입니다.

### 학습 목표

- Task Sizing 이 가장 단순하고도 효과적인 Context 관리 기법인 이유를 **설명합니다**
- 새 대화를 시작해야 하는 4 가지 신호를 **구별합니다**
- `/clear` 와 `/compact` 중 언제 무엇을 쓸지 **판단합니다**

## 긴 대화 한 통을 끝까지 끌고 가고 있다면

오전에 터미널을 하나 열어 로그인 API 를 붙이기 시작했다고 해보겠습니다. API 가 붙자 이어서 에러 메시지를 손보고, 같은 대화에서 테스트 코드를 쓰고, 내친김에 관련 파일 리팩터링까지 시작합니다. 오후가 되면 한 대화 안에 수천 줄의 메시지와 파일 수십 개의 도구 응답이 쌓여 있습니다.

이때쯤 이상한 일이 생깁니다. 분명 CLAUDE.md 에 적어둔 커밋 규칙을 Claude 가 무시하고, 아까 고친 파일을 다시 만져 달라고 하면 엉뚱한 함수를 건드립니다. 같은 수정을 두 번 요청해도 반영되지 않습니다. *모델을 잘못 골랐나, 지시를 잘못 썼나* 싶지만 원인은 둘 다 아닙니다 — **대화가 너무 길어진 것** 입니다.

앞 레슨의 Smart Zone · Dumb Zone 이 그대로 동작하는 중입니다. 오전에 올려둔 규칙은 어느새 Context 한가운데로 밀려가 놓치고, 수천 줄의 도구 응답이 주의력을 빼앗고 있습니다. CLAUDE.md · Memory 로 *무엇을 남길지* 는 풀었지만, *대화 자체의 길이* 는 아직 풀지 못한 상태입니다.

## Task Sizing: 한 잔이 넘치기 전에 새 잔으로

![lesson-04-task-sizing-cups](./attachments/lesson-04-task-sizing-cups.png)

2 리터짜리 물을 한 컵에 다 붓는 사람은 없습니다. 컵 한 잔이 찰 만큼 따르고, 비우고, 다시 따릅니다. **Task Sizing** 은 이 단순한 동작을 대화에 적용한 것입니다. 한 대화에 여러 작업을 몰아 붓지 않고, 한 잔만큼의 작업을 끝내면 새 대화를 시작합니다.

문제는 이 컵이 체감보다 훨씬 작다는 것입니다. Lesson 01 에서 본 것처럼 System Prompt · 도구 정의 · MCP 스키마 · CLAUDE.md 가 이미 컵의 상당 부분을 차지해, 사용자가 실제로 쓸 수 있는 공간은 좁습니다. *컵은 작다* 는 전제에서 시작해야 합니다.

### 한 잔의 자연스러운 크기 — 수정 → 테스트 → 커밋

좋은 한 잔의 단위는 **수정 → 테스트 → 커밋** 한 사이클입니다. 코드를 바꾸고, 테스트로 검증하고, 통과하면 커밋합니다. 이 사이클이 하나의 Context 안에서 완결되면 Claude 는 시작부터 끝까지 Smart Zone 에 머뭅니다.

반대로 한 대화에서 기능 A 를 만들고, 이어서 기능 B 를 디버깅하고, 기능 C 의 테스트까지 쓰면, A 의 도구 응답과 B 의 디버깅 로그가 컵을 점령해 C 를 시작할 때는 Claude 가 이미 Dumb Zone 에 있습니다. Task Sizing 을 무시한 대가는 느려진 응답이 아니라 **틀린 응답** 입니다.

> 메시지를 보내기 전마다 스스로 물어보세요 — *"이게 같은 잔에 담길 작업인가, 새 잔에 담길 작업인가?"*

## 새 잔으로 옮겨야 한다는 4 가지 신호

"언제 끊을지" 를 판단하는 것이 실전에서 가장 어렵습니다. 네 신호 중 하나라도 보이면 새 대화가 답입니다.

- **작업의 종류가 바뀔 때** — 프론트엔드 구현을 끝내고 백엔드 API 로 넘어간다면, 프론트엔드 코드와 도구 응답은 다음 작업에 필요 없습니다. 그대로 두면 주의력만 나눠 갖습니다
- **응답 품질이 눈에 띄게 떨어질 때** — 같은 수정을 두 번 요청했는데 반영이 안 됐다면 Dumb Zone 에 진입했거나 실패가 누적된 상태입니다. 어느 쪽이든 이어가는 것보다 새로 시작하는 편이 빠릅니다
- **커밋이 막 끝났을 때** — 커밋은 작업의 자연스러운 경계선입니다. 결과물이 이미 코드와 커밋 메시지에 남았으므로, 그 작업의 Context 를 더 붙잡고 있을 이유가 없습니다
- **끊을지 말지 망설일 때** — 고민된다는 건 이미 한 잔이 차오른다는 신호입니다. 새 대화가 거의 항상 더 나은 선택입니다

## /clear 와 /compact — 두 가지 잔 비우기

새 터미널을 여는 대신 현재 세션 안에서 컵을 비우는 두 명령어가 있습니다.

### /clear: 크기와 흐름을 동시에 리셋

`/clear` 는 Context 를 완전히 비웁니다. 대화 기록 · 도구 응답 · 누적된 정보가 모두 사라집니다. CLAUDE.md 는 System Prompt 의 일부라 그대로 유지되고, Memory 도 세션 시작 시점처럼 다시 로드됩니다. 사실상 새 대화를 여는 것과 같습니다.

`/clear` 가 단순히 크기만 리셋하는 게 아니라는 점이 핵심입니다. **대화의 흐름도 함께 끊깁니다.** 드라마에서 주인공이 계속 실패하는 장면이 이어지면 다음 장면에서도 실패할 것 같은 분위기가 생기는 것처럼, 수정 요청이 빗나가는 대화가 쌓이면 Claude 도 *"이 대화에서는 잘 안 되는 중"* 이라는 흐름을 이어갑니다. 일부러 그러는 게 아니라 지금까지의 대화 패턴에 맞춰 응답하는 것입니다. `/clear` 는 이 흐름을 끊고 깨끗한 상태에서 다시 시작하게 합니다.

<Callout type="idea" title="두 번 고쳐도 안 고쳐지면 `/clear`">
세 번째 시도를 Dumb Zone 에서 밀어붙이는 것보다 Smart Zone 에서의 첫 번째 시도가 성공 확률이 훨씬 높습니다. 같은 프롬프트를 새 대화에 그대로 붙여 넣는 것만으로도 결과가 달라지는 경우가 많습니다.
</Callout>

### /compact: 요약으로 Context 를 압축

`/compact` 는 대화 기록을 요약해 토큰 수를 줄입니다. 대화의 골자는 남으므로 *같은 작업을 계속 이어가야 하는데 Context 가 꽉 찰 때* 에 쓰입니다. 단, 요약 과정에서 **"아까 A 를 시도했다가 B 로 바꿨다"** 같은 미묘한 맥락이 사라져 같은 실수를 반복할 위험이 남습니다.

| 명령어 | 동작 | CLAUDE.md · Memory | 잃는 것 | 추천 상황 |
|---|---|---|---|---|
| `/clear` | 완전 초기화 | 유지 | 전체 맥락 | 작업 전환 · 품질 저하 · 커밋 완료 |
| `/compact` | 요약으로 압축 | 유지 | 뉘앙스 | 끊기 어려운 긴 작업을 이어가야 할 때 |

기본 선택은 `/clear` 입니다. `/compact` 는 *끊을 수 없는 긴 작업 하나를 억지로 이어가야 할 때* 에만 꺼내는 예외 도구에 가깝습니다.

## 핵심 포인트 정리

1. **대화 자체를 짧게 유지하기**: CLAUDE.md · Memory 가 *무엇을 남길지* 를 풀었다면, Task Sizing 은 *대화를 얼마나 길게 끌지* 를 푸는 기둥입니다. 한 잔이 넘치기 전에 새 잔으로 옮깁니다
2. **한 잔의 단위는 수정 → 테스트 → 커밋**: 한 사이클이 하나의 Context 에서 완결되면 시작부터 끝까지 Smart Zone 에 머뭅니다. 4 가지 신호 — 작업 전환 · 품질 저하 · 커밋 완료 · 망설임 — 중 하나라도 보이면 끊습니다
3. **기본은 `/clear`, 예외는 `/compact`**: `/clear` 는 크기뿐 아니라 실패의 흐름까지 리셋합니다. `/compact` 는 뉘앙스를 잃으므로, 끊을 수 없는 긴 작업을 어쩔 수 없이 이어가야 할 때만 씁니다

## FAQ

- **Q: 대화를 너무 자주 끊으면 매번 배경을 다시 설명해야 하는 것 아닌가요?**
  - A: CLAUDE.md 와 Memory 가 새 대화에서도 자동 로드되므로 프로젝트 배경과 개인 선호는 매번 설명하지 않아도 됩니다. 다시 설명하는 비용보다 Dumb Zone 에서 작업하면서 흘리는 비용이 훨씬 큽니다

- **Q: `/clear` 뒤에 이전 대화의 결과물을 참조하고 싶으면 어떻게 하나요?**
  - A: 이미 커밋된 결과물은 파일에 남아 있으므로 Claude 가 직접 읽으면 됩니다. 아직 코드에 반영되지 않은 논의는 한두 줄로 요약해 새 대화의 첫 메시지에 붙입니다

- **Q: `/compact` 만 계속 쓰면 안 되나요?**
  - A: 요약 과정에서 *"A 를 시도했다가 충돌 나서 B 로 바꿨다"* 같은 실패 맥락이 사라집니다. Claude 가 같은 실수를 다시 저지르기 쉬워지므로, 깨끗한 시작이 부분 요약보다 대체로 결과가 좋습니다
