## Overview

Chapter 03의 앞 세 레슨에서 Context Window의 구조, CLAUDE.md로 쓰는 팀 매뉴얼, Memory로 남기는 개인 노트까지 세웠습니다. 남은 질문은 하나입니다 — **하나의 대화를 얼마나 길게 이어갈까.** 답은 몇 년 사이에 크게 바뀌었고, 이번 레슨은 그 변화와 실전 신호를 정리합니다.

### 학습 목표

- 예전과 지금의 Task Sizing 기준이 어떻게 달라졌는지 **이해합니다**
- 새 대화를 시작해야 하는 두 가지 순간을 **판단합니다**
- `/clear` 와 `/compact` 를 언제 쓸지 **구분합니다**

## 얼마나에서 무엇을·어떻게로

![lesson-04-task-sizing-cups](./attachments/lesson-04-task-sizing-cups.png)

**예전.** Lost in the Middle이 지금보다 훨씬 심했습니다. 수천 줄짜리 대화가 쌓이면 중반부에 적어둔 지시가 통째로 사라지곤 했고, 안전한 권고는 단순했습니다 — **커밋마다 새 대화.** 하나의 대화에서 기능 A를 만들고 이어서 기능 B를 디버깅하면, B를 다룰 때쯤엔 A의 도구 응답이 주의력을 대부분 가져가 버렸습니다.

**지금.** 같은 기능 안에서 여러 파일·여러 커밋을 오가는 작업은 긴 대화 안에서도 비교적 잘 이어집니다. *"결제 API 연동 하나를 스펙 논의 → 구현 → 에러 처리 → 재시도 로직 → 테스트 까지 같은 대화에서 끌고 간다"* 같은 흐름은 이제 한 잔에 충분히 담깁니다. 예전 규칙을 그대로 적용해 매 커밋마다 `/clear` 를 하면 같은 오리엔테이션을 반복하느라 오히려 비효율입니다.

그럼에도 Task Sizing이 사라지지 않은 이유는, 긴 대화가 문제가 되는 **방식이 바뀌었기** 때문입니다. 예전에는 *양*이 문제였다면, 지금은 **무엇을 담았나·어떻게 담았나**가 문제입니다.

- **무엇** — 성격이 다른 일을 같은 잔에 부으면 완성 기준이 끌려 옵니다
- **어떻게** — 같은 잔에서 실패가 반복되면 그 실패 톤이 다음 답변까지 이어집니다

다음 두 섹션에서 각각 살펴봅니다.

## 성격이 다른 일을 이어 붙이면

프론트엔드 로그인 UI를 완성한 뒤, 같은 대화에서 이어서 백엔드 세션 토큰 만료 로직을 고쳐 달라고 해 봅니다. 모델은 잘 따라오는 듯 답하지만, 결과물을 자세히 보면 작은 어긋남이 섞여 있습니다. 프론트에서 쓰던 네이밍이 슬그머니 백엔드에 끌려 들어오거나, UI 작업의 느슨했던 테스트 기준이 결제 로직에까지 그대로 적용됩니다.

모델이 중간 정보를 *놓치는* 게 아니라, 앞서 담았던 잔의 *맛*이 새 음료에 남아 있는 상황입니다. 특히 끈질긴 것은 **완성 기준**입니다. 코드 스타일이나 네이밍은 한 번 지적하면 교정되지만, 이 기준은 이전 작업의 관성에 계속 끌려갑니다.

<Callout type="info" title="완성 기준이란?">
Claude가 대화 맥락에서 속으로 잡는 *"이 정도면 됐다"*의 선입니다. 사용자가 *"일단 뼈대만 만들어보자"*라고 말하면 Claude는 *"돌아가면 됨"* 수준에서 마무리하고, *"이번 주 안에 배포할 거야"*라고 말하면 엣지케이스·에러 복구까지 챙깁니다. 명시하지 않아도 대화의 톤에서 자동으로 읽어냅니다.
</Callout>

> UI에서는 눈으로 확인하면 충분했던 검증이, 결제 로직에서도 *"화면상 동작하니까 괜찮겠지"*로 이어집니다.

### 도메인이 같아도 생긴다

흔히 *"도메인이 바뀔 때 끊어라"*로 요약하지만, 실제로는 도메인보다 **완성 기준**이 더 일반적인 신호입니다. 같은 React 안에서도 프로토타입 단계와 프로덕션 릴리스 전 단계의 완성 기준은 다릅니다. 프로토타입에선 *"일단 돌아가면 됨"*이 기준이었다가, 같은 대화에서 프로덕션 기준으로 넘어가면 이전의 느슨함이 그대로 따라옵니다.

한 잔을 비우고, *지금 적용할 완성 기준*을 명시적으로 꺼내 담는 것이 안전합니다.

## 같은 실수가 두 번 반복되면

같은 수정 요청이 두 번 빗나간 상황을 떠올려 보세요. *"여기서 X를 바꿔달라"* 고 했는데 엉뚱한 곳을 고치고, 다시 설명해도 또 놓칩니다. 세 번째 시도에서 모델은 점점 방어적이고 조심스러워집니다. 대답 앞에 경고를 먼저 붙이거나, 작은 수정만 시도하다 물러서는 식입니다.

드라마에서 주인공이 연속 실패하는 장면이 이어지면 다음 장면도 실패 분위기가 깔리듯, 대화에 쌓인 **실패의 톤**이 다음 답변의 신뢰도를 스스로 낮춥니다. Claude는 *"이 대화에서는 잘 안 되는 중"*이라는 분위기에 맞춰 답을 냅니다.

### 문제는 대화의 흐름

세 번째 시도를 같은 대화에서 밀어붙이는 것보다, 같은 프롬프트를 새 대화에 그대로 붙여 넣는 편이 풀릴 확률이 높습니다. 모델이 이 문제를 못 푸는 게 아니라, **이 대화의 궤적이 답을 작게 만들고 있기** 때문입니다.

<Callout type="idea" title="두 번 빗나가면 /clear">
세 번째 시도를 실패 흐름 위에서 밀어붙이지 말고, 같은 프롬프트를 새 대화에 그대로 붙여 넣어 보세요. 같은 모델이 다른 답을 내놓는 경우가 많습니다.
</Callout>

## 새 대화를 여는 두 가지 순간

앞의 두 축을 실전 신호 두 가지로 옮기면 다음과 같습니다.

- **완성 기준이 바뀔 때 (가장 강한 신호)** — 프로토타입 → 프로덕션, UI → 결제, 기능 구현 → 성능 튜닝처럼 *"이 정도면 됐다"*의 기준선이 달라지는 순간. 도메인 전환보다 더 일반적이고, 이전 작업의 관성이 가장 크게 남는 지점입니다
- **같은 수정이 두 번 빗나갔을 때** — 세 번째 시도는 대부분 모델의 문제가 아니라 *대화의 흐름*이 만든 결과입니다. 같은 프롬프트를 새 대화에 그대로 붙여 넣는 것만으로 결과가 달라지는 경우가 많습니다

## /clear와 /compact

새 터미널을 여는 대신 현재 세션 안에서 잔을 비우는 두 명령어가 있습니다.

### /clear: 완전 초기화

`/clear` 는 대화 기록·도구 응답·누적된 정보를 전부 비웁니다. 세션 시작 시점부터 실려 있던 CLAUDE.md와 Memory는 그대로 남아, 사실상 새 대화를 여는 것과 같은 상태가 됩니다.

`/clear` 의 힘은 크기뿐 아니라 **대화의 흐름까지 리셋된다**는 점입니다. 실패가 쌓인 대화에서 `/clear` 는 궤적 자체를 끊어 깨끗한 상태에서 다시 시작하게 합니다.

### /compact: 요약으로 압축

`/compact` 는 대화 기록을 요약해 토큰 수를 줄입니다. 대화의 골자는 남기므로 *같은 작업을 계속 이어가야 하는데 Context가 꽉 찰 때*에 쓰입니다. 단, 요약 과정에서 *"A를 시도했다가 충돌 나서 B로 바꿨다"* 같은 실패 경위가 사라져, 같은 실수를 반복할 위험이 남습니다.

| 명령어 | 동작 | 잃는 것 | 언제 쓰나 |
|---|---|---|---|
| `/clear` | 완전 초기화 | 전체 맥락 | 완성 기준 전환 · 실패 흐름 리셋 |
| `/compact` | 요약으로 압축 | 실패 경위·뉘앙스 | 끊을 수 없는 긴 작업을 이어가야 할 때 |

기본 선택은 `/clear` 입니다. `/compact` 는 *끊을 수 없는 긴 작업을 억지로 이어가야 할 때* 에만 꺼내는 예외 도구입니다.

## 핵심 포인트 정리

1. **기준이 양에서 무엇·어떻게로 이동**: 모델이 좋아지면서 양 자체는 덜 문제가 됐습니다. 대신 *무엇을 담았나(완성 기준)*와 *어떻게 쌓였나(실패 흐름)*가 새 기준입니다
2. **완성 기준이 바뀌는 순간이 가장 강한 신호**: 도메인 전환보다 *"이 정도면 됐다"*의 기준선이 달라지는 순간이 더 위험합니다
3. **기본은 `/clear`, 예외는 `/compact`**: `/clear` 는 Context 크기뿐 아니라 실패 흐름까지 리셋합니다. `/compact` 는 뉘앙스를 잃으므로 끊을 수 없는 긴 작업을 어쩔 수 없이 이어갈 때만 씁니다

## FAQ

- **Q: 예전에 배운 "커밋마다 새 대화" 규칙은 이제 틀린 건가요?**
  - A: 틀렸다기보다 *과보정*에 가깝습니다. 모델이 좋아진 지금은 같은 기능 안에서의 여러 커밋을 한 대화로 이어도 문제가 없습니다. 커밋 자체는 끊는 자리가 아니라, *다음 작업이 같은 완성 기준인가*를 점검하는 체크포인트로 쓰면 충분합니다

- **Q: 한 대화를 어디까지 이어갈 수 있나요?**
  - A: 정해진 토큰 수가 아니라 *완성 기준이 유지되는 범위*가 기준입니다. 같은 기준에 머무는 한 대화가 길어져도 품질이 크게 떨어지지 않는 것이 최신 모델의 경향입니다. 단, 비용과 레이턴시는 길이에 비례해 늘어나므로 실용적으로는 몇 시간 단위로 한 번 끊는 감각이 편합니다

- **Q: `/compact` 만 계속 쓰면 안 되나요?**
  - A: 요약 과정에서 *"A를 시도했다가 충돌 나서 B로 바꿨다"* 같은 실패 경위가 사라집니다. 같은 실수를 다시 저지르기 쉬워지므로, 깨끗한 시작이 부분 요약보다 대체로 결과가 좋습니다
