## LLM은 빈칸을 추측으로 메우는 확률 기계

Part 1은 "AI가 왜 자신있게 틀리는가"에서 시작했습니다. LLM은 "맞는 답"이 아니라 **그럴듯한 다음 단어**를 예측하는 시스템이고, 비어 있는 자리는 확률적으로 그럴듯한 값으로 메웁니다. 이 원리가 Part 1 전체에 깔려 있습니다.

### 구조적 한계와 그 우회

- **Hallucination·Knowledge Cutoff는 줄일 수는 있지만 없앨 수 없습니다**: "그럴듯한 다음 단어"라는 원리 자체의 부산물이기 때문에, 정답과 오답이 똑같은 자신감으로 나옵니다
- **Tool은 한계를 지우지 않고 우회합니다**: 예측의 근거를 머릿속 기억에서 실제 파일·검색 결과로 바꿔줄 뿐, LLM이 확률로 답을 만든다는 본질은 그대로입니다
- **Agent = Tool + Loop + 자율 판단**: 한 번 쓰고 끝나면 LLM + Tool, 결과를 보고 다음 행동을 스스로 정하며 반복하면 Agent입니다. 능력의 차이가 아니라 **실행 구조**의 차이입니다

### 코딩 도구의 3단계 진화와 Claude Code

- **자동완성 → 대화형 편집 → Agentic**: 단계를 거칠수록 AI가 보는 맥락이 넓어지고 자율성이 커집니다. Agentic 코딩은 방향만 주면 AI가 파일을 읽고·수정하고·실행까지 자율 수행하는 단계입니다
- **Claude Code = Agentic 코딩 + Terminal-native + 프로그래밍 가능**: 어디서든 실행되고, 동작 방식 자체를 프로젝트 파일(CLAUDE.md·Hooks·Skills·Subagent·MCP)로 설계합니다. 프로그래밍 가능 축은 Part 2에서 본격적으로 다룹니다

## Claude Code — Agent를 매일 다루는 판

Agent를 하루 종일 쓰려면 도구의 감각부터 몸에 익어야 합니다. Part 1 Chapter 02는 이 감각의 네 축과 모델 전략을 깔았습니다.

### 매일 부딪히는 네 축

- **세션 — 한 대화의 수명**: `claude`·`--continue`·`--resume` 로 세션을 열고 `/exit` 로 닫습니다. 대화가 길어지면 `/clear` 로 비우고, `/compact` 는 요약 과정에서 뉘앙스가 사라져 되도록 피합니다
- **특수 입력 — `!` `@` `/`**: 입력창 첫 문자가 모드 전환 스위치입니다. `!` 는 쉘 명령을 바로 실행하고, `@` 는 파일을 Context에 첨부하며, `/` 는 슬래시 명령을 호출합니다. 긴 입력은 `Ctrl+J`, 이미지는 `Ctrl+V` 로 붙입니다
- **권한 — 여섯 모드**: `Shift+Tab` 으로 순환합니다. `auto` 는 분류기가 매 행동을 사전 검토하는 자동 모드이고, `bypassPermissions` 는 안전장치가 없는 마지막 수단입니다
- **되돌리기 — Rewind와 Checkpoint**: `Esc+Esc` (또는 `/rewind`)로 마지막 AI 행동을 통째로 되돌리고, Checkpoint로 여러 단계 전의 상태까지 한 번에 이동합니다

### 모델과 판단의 분업

- **Opus로 계획, Sonnet으로 구현, Haiku로 빠른 작업**: 설계·복잡한 추론은 Opus, 일반 구현·리뷰·테스트는 Sonnet, 단순 수정·짧은 질문은 Haiku로 나눠 쓰면 품질과 비용이 함께 잡힙니다. 모델 전환은 세션 안에서도 가능하며 `/model` 로 바꿔도 Context는 그대로 이어집니다
- **판단의 분업 — 개발자는 방향, AI는 다음 단계**: 개발자는 "무엇을 만들지·어떤 제약을 둘지·이 결과가 맞는지"를 정하고, AI는 "그걸 만들기 위해 지금 뭘 해야 하는지"를 판단합니다. 이 분업이 자연어 한 줄로 읽기 → 이해 → 수정 → 커밋 사이클을 돌리는 바탕입니다

## 작은 책상을 잘 관리하는 법

<SmartDumbZone />

Claude의 작업 책상(**Context Window**)은 생각보다 좁습니다. 같은 모델·같은 지시여도 책상이 어떻게 채워져 있느냐에 따라 결과가 갈립니다.

### Context Window의 두 함정

- **Smart Zone·Dumb Zone (위치 편향)**: Context 초반·후반부에서 주의력이 가장 높고 **중간 구간의 정보를 놓치는 경향**이 강합니다. 업계에서는 이 현상을 Lost in the Middle 이라 부릅니다
- **지침의 저주**: 지침이 많아질수록 개별 지침의 준수율이 떨어집니다. 해법은 더 자세히 쓰는 것이 아니라 **지금 이 작업에 필요한 것만 주는 것** 이고, 이 원칙이 Claude Code의 Rules·Skills·Sub-agents·CLAUDE.md에 각각 다른 전략으로 구현돼 있습니다

### 영구 지식의 분담 — CLAUDE.md와 Memory

- **CLAUDE.md — 내가 쓰는 팀 매뉴얼**: 매 세션 자동 로드되고, 판단 기준은 "모델이 코드에서 스스로 찾을 수 없는가"입니다. 기술 스택·디렉토리·명령어는 빼고, 아키텍처 결정 이유·팀 워크플로우·제약사항만 남깁니다. 300줄 미만 유지가 권장입니다
- **Memory — Claude가 쓰는 관찰 노트**: 대화 중 감지한 개인 선호를 자동 저장해 다음 세션에 불러옵니다. 로컬 전용이라 팀원과 공유되지 않고, 충돌 시 CLAUDE.md가 이깁니다. CLAUDE.md가 다루지 않는 **나만의 빈칸**을 채우는 자리입니다

### Task Sizing — 대화를 언제 끊나

- **기준이 **양**에서 **무엇을·어떻게 담았나**로 바뀌었습니다**: 모델이 좋아지면서 같은 기능의 여러 커밋을 한 대화로 이어도 품질이 유지됩니다. 대신 **완성 기준**과 **실패 흐름**이 새 변수가 됐습니다
- **새 대화를 여는 두 순간**: ① "이 정도면 됐다"의 기준선이 달라질 때 (프로토타입 → 프로덕션, UI → 결제 등), ② 같은 수정이 두 번 빗나갔을 때. 세 번째 시도는 같은 프롬프트를 새 대화에 그대로 붙이는 편이 풀릴 확률이 높습니다

## 계획이 먼저, 코드는 그 다음

Part 1의 마지막 장은 "빈 폴더에 대고 바로 코드를 시키면 왜 실패하나"에서 시작했습니다. 답은 첫 테마로 돌아옵니다 — AI는 빈칸을 추측으로 메우고, 개발자도 자신이 원하는 걸 처음부터 정확히 모릅니다.

### Plan Mode — 쓰기를 잠가 탐색을 강제

- **`Shift+Tab` 두 번으로 진입**: Edit·Write·Bash가 차단되고 Read·Grep·Glob만 열립니다. 첫 출근한 신입에게 "먼저 코드부터 읽어보고 오라" 고 말하는 것과 같은 효과입니다
- **탐색 → 계획 → 실행 이 한 번의 진입으로 이어집니다**: 탐색에서 쌓인 맥락이 실행 단계까지 그대로 따라가므로, 처음부터 "코드 짜줘"로 시작했을 때보다 결과가 정확해집니다
- **AI의 역질문이 가장 큰 가치입니다**: "완료한 항목은 보관할까요, 삭제할까요?" 같은 질문은 개발자가 스스로에게 물어본 적 없는 빈칸을 드러냅니다. 코드를 다 쓰고 되돌리는 것보다 훨씬 저렴합니다

### 빈칸의 두 자리 — 요구사항과 기술 스택

- **요구사항은 두 섹션으로 명시합니다**: **기능 목록**은 "사용자가 ~하면, ~한다" 형식으로 동작과 결과를 함께 적고, **범위 제한**은 "무엇을 만들지 않는가"로 AI가 "있으면 좋을 것 같아서" 덧붙이는 기능을 막습니다
- **AI 친화 스택 = 학습 데이터 밀도 + 코드 가시성**: Next.js는 학습 데이터가 두텁고, Shadcn은 컴포넌트 소스를 프로젝트 안에 복사해 AI가 직접 열어 읽고 판단합니다. 두 축 모두 AI의 빈칸을 줄이는 방향입니다

## 이어서 배울 내용

Part 2는 이 질문에서 시작합니다. 체크리스트를 테스트 코드로 변환하면 AI가 스스로 성공 기준을 읽으며 Loop를 돌고, 개발자는 단계를 하나하나 지시하는 대신 성공 기준만 주고 방법은 AI에게 맡기게 됩니다.

- What vs How — AI에게 일을 시키는 두 가지 방법
- 테스트 기반 검증 — 수동 체크리스트의 자동화
- Red Green Refactor — 성공 기준을 테스트로 변환하고 한 번에 하나씩 구현
- Task 시스템 — 대화가 끊겨도 이어가는 작업 진행
