## Overview

지난 레슨에서 세션·입력·권한·되돌리기 네 축을 익혔습니다. 도구는 준비됐지만 Claude가 실제 코드 앞에서 어떻게 움직이는지는 직접 부딪쳐야 감이 옵니다.

이번 레슨은 작은 Todo 앱을 받아 **코드 읽기 → 한 파일 깊이 보기 → README 수정 → git 커밋**까지 네 번의 대화로 첫 개발 사이클을 끝냅니다. 매 단계마다 Claude가 어떤 도구를 쓰고 어떤 권한이 필요한지 관찰합니다.

### 학습 목표

- Claude가 답을 추측하지 않고 파일을 직접 읽으며 답하는 모습을 **관찰합니다**
- `@` 로 한 파일에 집중시켜 답변 정확도를 **올립니다**
- 쓰기 작업에서 권한 승인 흐름을 **직접 거칩니다**
- 자연어로 git 커밋을 지시하고 결과를 `!` 로 **확인합니다**

### 시작하기 전 확인사항

- Claude Code 설치·로그인 완료 (`claude --version` 으로 확인)
- VS Code 또는 시스템 터미널 사용 가능

이번 실습은 강의에서 제공하는 Todo 앱 저장소를 clone 해서 진행합니다.

![vscode-welcome](/learn/starting-conversations/getting-started/vscode-welcome.png)

VS Code를 열고 Welcome 탭에서 **Clone Git Repository...**를 클릭합니다. 상단 입력창에 다음 URL을 붙여넣고 Enter를 누릅니다.

```
https://github.com/toy-crane/todo-tutorial.git
```

저장 위치를 고르면 VS Code가 자동으로 클론하고 폴더를 열어줍니다. 통합 터미널을 열고(`` Ctrl+` ``) 실습 시작 브랜치로 전환합니다.

```shell
git checkout ch02-03
```

<Callout type="info" title="강의 Q&A 도우미가 이미 등록돼 있어요">
이 저장소에는 강의 Q&A 도우미(`/qna`)가 미리 등록돼 있습니다. 강의 내용이 헷갈릴 때 Claude Code 입력창에 한국어로 그냥 물어보시면 됩니다. "Skill 이 뭐였지?" 처럼 강의에서 다룬 개념을 묻거나, "Context Window 다시 설명해줘" 같은 질문에 자동으로 활성화됩니다. 이 도우미가 어떻게 들어왔는지는 Chapter 06 의 "기존 Skill 가져다 쓰기" 레슨에서 다시 다룹니다.
</Callout>

이번 실습은 네 번의 대화로 진행됩니다. 읽기에서 쓰기로 한 단계씩 옮겨가며, Claude가 어떤 도구를 사용하고 어떤 권한이 필요한지 관찰하게 됩니다.

<FirstConversationFlow />

VS Code 통합 터미널에서 Claude Code를 실행합니다.

```shell
claude
```

VS Code가 프로젝트 폴더에서 터미널을 열기 때문에 별도로 `cd` 할 필요가 없습니다.

## Step 1 — "이 프로젝트가 뭐하는 프로젝트야?"로 코드 읽히기

Claude Code 입력창에 다음을 입력하고 Enter를 누릅니다.

> 이 프로젝트가 뭐하는 프로젝트야?

Claude는 답을 추측하지 않습니다. README.md, package.json, 주요 소스 파일을 **직접 열어 읽고** 그 결과를 바탕으로 답합니다. 터미널에서 어떤 파일을 여는지 실시간으로 확인할 수 있습니다.

Chapter 01에서 본 **Tool Use**가 실제로 작동하는 장면입니다. 읽기 작업은 승인 없이 자유롭게 수행됩니다.

<Callout type="info" title="프로젝트 규모에 따라 탐색 시간이 다릅니다">
작은 프로젝트는 몇 초면 충분하지만, 큰 프로젝트는 더 많은 파일을 훑어야 합니다. Claude가 파일을 읽는 중이라면 `Ctrl+C` 로 중단하지 말고 기다려 주세요.
</Callout>

## Step 2 — @ 로 한 파일에 집중시키기

전체를 훑었으니 이번에는 한 파일을 깊게 봅니다. 지난 레슨에서 익힌 `@` 접두어를 씁니다.

> @app/page.tsx이 파일이 하는 일을 설명해줘

`@` 를 입력하면 파일 경로 자리에서 탭 자동완성이 동작합니다. 지정한 파일이 Claude의 Context로 직접 들어가기 때문에, "page.tsx 설명해줘"처럼 검색을 시키는 쪽보다 답이 더 정확하고 빠릅니다.

Claude는 함수의 역할, 데이터 흐름, 다른 파일과의 관계를 정리해 답합니다.

## Step 3 — 쓰기엔 승인이 따른다 (README 수정)

여기서부터 Claude가 파일을 **수정** 합니다. 읽기와 달리 쓰기는 승인을 거칩니다.

> README의 Contributors 섹션에 "내 이름 - Frontend Developer"를 추가해줘

수정을 시도하면 권한 프롬프트가 나타납니다.

```
Claude wants to edit README.md

Allow?  [y] Yes  [a] Always allow  [n] No
```

`y` 로 이번 한 번만 허용해 봅니다. 수정이 끝나면 변경 내용이 **diff**로 표시됩니다. 추가된 줄은 초록색, 삭제된 줄은 빨간색입니다.

결과가 마음에 들지 않으면 `Esc+Esc` (또는 `/rewind`)로 되돌린 뒤 더 구체적인 지시로 다시 요청합니다. 일반적으로 **구체적인 지시일수록 원하는 결과에 가깝습니다.**

## Step 4 — 자연어로 git 커밋

마지막은 변경사항 저장입니다.

> 변경사항 커밋해줘

Claude는 `git add` 와 `git commit` 을 실행합니다. 셸 명령어 실행에도 승인이 필요합니다.

```
Claude wants to run: git add README.md

Allow?  [y] Yes  [a] Always allow  [n] No
```

`y` 로 허용하면 Claude가 변경 내용을 분석해 커밋 메시지를 자동으로 작성합니다. 결과는 `!` Bash 모드로 빠르게 확인할 수 있습니다.

> ! git log --oneline -1

```
a1b2c3d docs: add contributor to README
```

네 번의 대화만으로 프로젝트 이해부터 파일 수정과 git 커밋까지 끝났습니다.

## 원칙 요약 — 읽기 → 이해 → 수정 → 저장

네 단계의 순서는 우연이 아닙니다. 평소 개발자가 매일 도는 사이클을 그대로 따라갑니다.

1. **읽기 (Step 1·2)** — 무엇이 있는지부터 본다. 추측 대신 실제 파일을 근거로
2. **이해** — 읽은 내용을 바탕으로 무엇을 어떻게 바꿀지 결정. 사람이 판단하는 자리
3. **수정 (Step 3)** — 정해진 변경을 코드에 반영. 승인 흐름이 안전망 역할
4. **저장 (Step 4)** — 변경을 커밋해 다음 단계로 넘김. 흐름의 마디

자연어 한 줄이 사이클의 각 단계를 트리거하지만, **"무엇을 만들지·결과가 맞는지"**의 판단은 여전히 사람의 몫입니다. Chapter 01에서 본 "판단의 분업"이 실전에서 도는 모습입니다.

## 핵심 포인트 정리

1. **Claude는 추측하지 않고 파일을 직접 읽습니다**: 답이 막연해 보일 때도 내부에서는 Tool Use가 돌고 있습니다
2. **읽기는 자유, 쓰기·셸 실행은 승인이 필요합니다**: diff로 확인하고 `Esc+Esc` 로 되돌릴 수 있어 처음 써도 안전합니다
3. **`@` 로 파일을 집중시키면 정확도가 올라갑니다**: 검색하라고 시키는 것보다 직접 첨부하는 쪽이 빠르고 정확합니다
4. **자연어 한 줄이 git 커밋까지 끌고 갑니다**: 단, 결과의 맞고 틀림은 사람이 판단해야 합니다

<Callout type="idea" title="Claude Code에 대해 모르는 게 있으면 Claude Code에게 물어보세요">
Claude Code의 기능·설정·단축키가 궁금할 때 외부 ChatGPT나 Claude 웹에 물어보지 마세요. 같은 세션 안에서 직접 질문하는 편이 가장 빠릅니다. Claude Code는 현재 프로젝트에 접근할 수 있어 상황을 파악하고 직접 조치까지 가능합니다. 내부적으로 `claude-code-guide` 라는 전문 서브에이전트가 Claude Code 자체에 대해 정확하게 답합니다 — 명시적으로 부르고 싶으면 입력창에 `@claude-code-guide` 처럼 `@` 뒤에 에이전트 이름을 입력하면 바로 호출됩니다.
</Callout>

## FAQ

- **Q: Claude가 파일을 너무 많이 읽어 시간이 오래 걸리면 어떻게 하나요?**
  - A: 탐색 범위를 좁혀 지시하면 됩니다. "src/api 폴더만 설명해줘"처럼 범위를 명시하면 훨씬 빠르게 답합니다

- **Q: 커밋 메시지를 직접 지정하고 싶어요.**
  - A: 구체적으로 지시합니다. `변경사항을 'feat: add contributor section' 메시지로 커밋해줘` 라고 하면 그 메시지가 그대로 사용됩니다. 매번 지정하기 번거롭다면 반복 작업을 자동화할 수 있습니다 (Chapter 06의 Custom Command)

- **Q: Claude가 잘못된 파일을 수정하려 하면 어떻게 막나요?**
  - A: 승인 단계에서 `n` 으로 거부합니다. 작업이 차단되고 Claude는 다른 방법을 시도하거나 다음 지시를 기다립니다. 이미 승인한 후라면 `Esc+Esc` 로 되돌립니다

## 이어서 배울 내용

네 번의 대화만으로 기본 개발 사이클을 돌았습니다. 다만 이 흐름은 **대화가 짧을 때만** 매끄럽습니다. 같은 세션을 길게 끌고 가면 어느 순간부터 Claude의 답이 흐릿해지고 같은 실수를 반복하는 현상이 나타납니다.

다음 Chapter에서는 이 현상의 원인과 해결 방법을 다룹니다.

- Context Window: AI가 한 번에 볼 수 있는 범위와 그 한계
- CLAUDE.md: 프로젝트 정보를 한 번만 적고 매번 자동 제공하기
- Memory: 대화가 끊겨도 학습한 내용을 유지하는 시스템
- Task Sizing: 대화를 끊는 기술로 품질 유지
