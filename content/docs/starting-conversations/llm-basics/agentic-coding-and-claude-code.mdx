## Overview

앞 레슨에서 Agent를 "Tool + Loop + 자율적 판단"이라는 실행 구조로 정의했습니다. 이 구조가 코딩 도구에 정면으로 들어오면 어떤 모양이 될까요?

이 레슨에서는 코딩 도구가 자동완성 · 대화형 편집을 거쳐 도달한 **Agentic 코딩**이 무엇인지 정리하고, 그 위에 **Claude Code**가 어떤 두 축을 더해 확장된 도구인지 살펴봅니다.

### 학습 목표

- Agentic 코딩이 자동완성 · 대화형 편집과 어떻게 다른지 이해합니다
- 앞 레슨의 Agent 공식(Tool + Loop + 자율 판단)이 실제 개발 작업에서 어떻게 구체화되는지 이해합니다
- Claude Code가 Agentic 코딩 위에 Terminal-native · 프로그래밍 가능이라는 두 축을 어떻게 더해 확장되는지 이해합니다

## Agentic 코딩: 지시 대신 위임하는 방식

<CodingToolEvolution />

**Agentic 코딩**은 "이 Todo 앱에 완료된 할 일을 숨기는 필터를 추가해줘"처럼 방향만 제시하면, AI가 관련 파일을 찾고 · 수정하고 · 테스트까지 자율적으로 돌리는 방식입니다. 손으로 조작하는 범위가 줄고, 판단을 맡기는 범위가 늘어납니다.

이것이 어디서 왔는지 이해하려면 코딩 도구가 어떻게 진화해 왔는지 먼저 봐야 합니다.

### 자동완성 → 대화형 편집 → Agentic, 진화의 3단계

| 단계 | 맥락 | 도구가 하는 일 | 대표 도구 |
|---|---|---|---|
| 1. 자동완성 | 현재 파일 | 다음 단어 제안 (수락/거부) | GitHub Copilot, Tabnine |
| 2. 대화형 편집 | 프로젝트의 여러 파일 | 요청을 받아 코드 편집 | Cursor, Windsurf, Copilot Chat |
| 3. Agentic | 프로젝트 전체 + 실행 환경 | 방향을 받아 파일 읽기·쓰기·명령어 실행을 자율 수행 | Claude Code, Codex, Gemini CLI |

단계를 거듭할수록 AI에게 주어지는 **맥락**이 넓어지고 **자율성**이 커집니다. 자동완성은 "지금 이 줄"만 보고, 대화형 편집은 "이 파일들"을 보고, Agentic은 "프로젝트 + 실제 실행 결과"까지 봅니다. 세 단계가 대체 관계는 아닙니다 — 한 줄 자리 채움에는 자동완성이, 짧은 수정에는 대화형 편집이 여전히 빠르고, Agentic은 여러 파일을 넘나드는 복잡한 작업에서 강점을 발휘합니다.

### 앞 레슨 공식을 코딩에 그대로 대입하면

앞 레슨에서 본 공식을 떠올려 봅시다.

> **Agent = Tool + Loop + 자율적 판단**

Agentic 코딩은 이 공식을 코딩 맥락에 그대로 대입한 것입니다.

- **Tool** — 파일 읽기/쓰기, 터미널 명령어 실행, 웹 검색. 개발자가 IDE · 터미널에서 하는 모든 행동
- **Loop** — 한 번의 요청 뒤에 탐색 → 수정 → 실행 → 확인을 여러 차례 반복
- **자율적 판단** — "다음에 어느 파일을 열지", "테스트가 실패했으면 어디를 고칠지"를 스스로 결정

Agentic 코딩은 새로운 원리가 아니라, 앞 레슨의 Agent 구조를 코딩이라는 영역에 대입한 결과입니다.

## 실제 개발 작업에서 무엇이 달라지나

앞 레슨에서 꺼냈던 버그 수정 시나리오를 떠올려 봅시다.

> "로그인 페이지에서 비밀번호 찾기 링크를 눌러도 아무 반응이 없어. 고쳐줘."

같은 요청을 각 단계의 도구에 던지면 어떻게 흘러갈까요?

### "버그 고쳐줘" 한 마디 뒤에서 벌어지는 루프

<BugFixLoop />

자동완성 도구라면 — 개발자가 해당 컴포넌트를 직접 찾아 열고, 문제 줄 근처에서 "다음 코드 제안"을 받는 수준까지만 돕습니다. 탐색과 실행은 전부 개발자의 몫입니다.

대화형 편집 도구라면 — 개발자가 원인을 직접 진단한 뒤 "이 함수에 onClick 핸들러 추가해줘"처럼 구체적인 지시를 내려야 수정이 일어납니다. 무엇을 고쳐야 하는지는 여전히 개발자가 판단합니다.

Agentic 코딩 도구라면 — "비밀번호 찾기 링크가 안 먹힌다"는 증상만 받아 다음을 스스로 반복합니다.

1. 파일 검색으로 "비밀번호 찾기" 관련 컴포넌트를 찾습니다
2. 링크 컴포넌트와 라우터 설정을 **읽고** 원인 후보를 좁힙니다
3. 수정안을 파일에 **씁니다**
4. 개발 서버 로그나 테스트를 **실행**해 결과를 **확인**합니다
5. 실패하면 다시 1~4를 반복합니다

개발자가 한 일은 "버그가 있다"는 방향 제시뿐입니다. "어디를 볼까, 어떻게 고칠까, 고친 뒤 뭘 확인할까"는 AI가 판단합니다.

### 개발자는 방향을 정하고, AI는 다음 단계를 판단한다

핵심 변화는 **판단의 분업**입니다.

- 개발자 — "무엇을 만들 것인가", "어떤 제약을 둘 것인가", "이 결과가 맞는가"
- AI — "그것을 만들기 위해 지금 뭘 해야 하는가"

이 분업이 성립하려면 AI가 프로젝트 전체의 파일을 읽고, 명령어를 돌리고, 실패 결과를 해석할 수 있어야 합니다. 자동완성 · 대화형 편집까지는 개발자가 계속 중재자 역할을 해야 했지만, Agentic 코딩에서는 그 중재자 역할이 줄어듭니다.

## Claude Code: 어디서든 실행되고, 동작까지 설계 가능한 Agent

Agentic 코딩 도구는 하나가 아닙니다. Claude Code, Codex, Gemini CLI가 모두 같은 범주에 들어갑니다. 그중 Claude Code는 다음 두 축 위에 서 있습니다.

> **Claude Code = Agentic 코딩 + Terminal-native + 프로그래밍 가능**

### Terminal-native: 편집기에 갇히지 않는다

Claude Code는 터미널에서 돌아가는 독립 CLI 도구입니다. 특정 IDE 안의 기능이 아니라, 터미널이 있는 곳이면 어디서든 호출할 수 있습니다.

- 로컬 개발 환경 — IDE를 안 띄우고 터미널만으로도 사용
- 원격 서버 — SSH로 붙은 서버 안에서 실행
- CI/CD 파이프라인 · 스크립트 · Docker 컨테이너 — 자동화 흐름 안에 끼워 넣기

VS Code · JetBrains 확장으로도 제공하긴 하지만, 본질은 "어디서든 돌아가는 CLI"입니다.

### 프로그래밍 가능: 동작 방식 자체를 코드로 정의

일반 코딩 도구는 정해진 기능만 쓸 수 있습니다. Claude Code는 **동작 규칙 자체**를 프로젝트 안에 파일로 정의할 수 있습니다.

- **CLAUDE.md** — 프로젝트 지침서. 코드 컨벤션 · 주의사항을 AI에게 상시로 전달
- **Hooks** — 작업 전/후에 자동으로 돌아가는 스크립트 (lint, 테스트, 차단 규칙)
- **Skills** — 반복 작업을 정리해둔 매뉴얼
- **Subagent** — "테스트 담당", "리뷰 담당"처럼 역할을 나눈 전문 AI
- **MCP** — 데이터베이스 · 외부 API 등 외부 시스템과의 연결 통로

이 도구들을 어떻게 쓰는지는 Chapter 03과 Chapter 06-08에서 하나씩 다룹니다. 지금은 "Claude Code는 정해진 기능 목록이 아니라, 내 프로젝트에 맞게 동작을 설계하는 판"이라는 점만 기억하면 됩니다.

### 왜 두 축이 중요한가

Terminal-native와 프로그래밍 가능은 "있으면 편한 부가 기능"이 아니라, Agentic 코딩의 가치를 실제 작업 환경에서 끌어내기 위한 축입니다.

- Agent가 **어디서든** 돌지 못하면, 자동화 흐름에 붙이기 어렵습니다
- 동작을 **내가 설계**하지 못하면, 프로젝트의 규칙을 AI에게 매번 반복 설명해야 합니다

두 축이 있으면 Claude Code는 범용 도구가 아니라 **내 프로젝트에 맞춰진 Agent**가 됩니다.

## 핵심 포인트 정리

1. **Agentic 코딩 = 앞 레슨의 Agent 공식을 코딩에 적용**: 파일 · 터미널 · 실행 환경을 Tool로 쓰고, 탐색·수정·실행을 Loop로 반복하며, 다음 단계를 스스로 판단합니다
2. **판단의 분업**: 개발자는 무엇을·왜 만드는지를 정하고, AI는 그걸 만드는 중간 단계를 판단합니다
3. **Claude Code = Agentic 코딩 + Terminal-native + 프로그래밍 가능**: 어디서든 실행되고, 동작 방식 자체를 프로젝트 안에 파일로 설계할 수 있는 Agent입니다

## FAQ

- **Q: Agentic 코딩이면 개발자가 필요 없어지나요?**
  - A: 코드를 **생성하는** 비용은 낮아졌지만, 무엇을 만들지 · 어떤 제약을 둘지 판단하는 일은 여전히 사람의 몫입니다. 아키텍처 선택, 엣지 케이스 판단, 결과 검증은 엔지니어링 경험이 있어야 가능합니다

- **Q: 대화형 편집 도구를 잘 쓰고 있는데, 굳이 Agentic으로 넘어가야 하나요?**
  - A: 꼭 그럴 필요는 없습니다. 짧은 수정이나 단일 파일 작업에는 대화형 편집이 더 빠릅니다. 여러 파일을 넘나들고 테스트까지 돌려야 하는 작업에서 Agentic이 결정적인 차이를 만듭니다

- **Q: CLAUDE.md, Hooks, Skills 같은 개념을 다 알아야 시작할 수 있나요?**
  - A: 아닙니다. Claude Code는 아무 설정 없이도 바로 돌아갑니다. 필요해질 때 하나씩 얹으면 됩니다
