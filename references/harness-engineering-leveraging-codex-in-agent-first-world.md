지난 5개월 동안 우리 팀은 한 가지 실험을 진행했다. **수동으로 작성한 코드가 단 한 줄도 없는** 소프트웨어 제품의 내부 베타를 개발하고 배포하는 것이었다.

이 제품은 내부 일일 사용자와 외부 알파 테스터를 보유하고 있다. 배포되고, 중단되고, 수정된다. 다른 점은 애플리케이션 로직, 테스트, CI 구성, 문서, 관찰성(observability), 내부 도구를 포함한 모든 코드 라인이 Codex에 의해 작성되었다는 것이다. 우리는 손으로 코드를 작성했을 때보다 약 1/10의 시간으로 이를 구축한 것으로 추정한다.

**인간은 방향을 잡고, 에이전트는 실행한다.**

우리는 의도적으로 이러한 제약을 선택했다. 엔지니어링 속도를 몇 배나 높이는 데 필요한 것을 구축하기 위해서였다. 결국 백만 줄의 코드가 되는 것을 몇 주 안에 배포해야 했다. 그러기 위해서는 소프트웨어 엔지니어링 팀의 주된 업무가 더 이상 코드를 작성하는 것이 아니라 환경을 설계하고, 의도를 명시하고, Codex 에이전트가 안정적으로 작업할 수 있는 피드백 루프를 구축하는 것으로 바뀔 때 무엇이 달라지는지 이해해야 했다.

이 글은 에이전트 팀으로 완전히 새로운 제품을 구축하면서 배운 것들에 관한 것이다. 무엇이 망가졌고, 무엇이 누적되었으며, 우리의 유일하게 진정 희소한 자원인 인간의 시간과 주의력을 어떻게 극대화할 수 있는지에 관한 이야기다.

## 빈 git 저장소에서 시작했다

첫 커밋은 2025년 8월 말 빈 저장소에 추가되었다.

초기 스캐폴드(저장소 구조, CI 구성, 포매팅 규칙, 패키지 매니저 설정, 애플리케이션 프레임워크)는 GPT-5를 사용하는 Codex CLI로 생성되었고, 기존 템플릿 소수로 안내되었다. 에이전트가 저장소에서 어떻게 작업해야 하는지 지시하는 초기 AGENTS.md 파일조차 Codex가 직접 작성했다.

시스템을 고정할 기존의 사람이 작성한 코드는 없었다. 처음부터 저장소는 에이전트에 의해 형성되었다.

5개월 후, 저장소는 애플리케이션 로직, 인프라, 도구, 문서, 내부 개발자 유틸리티에 걸쳐 약 백만 줄의 코드를 포함하게 되었다. 이 기간 동안 Codex를 운영하는 엔지니어 3명으로 구성된 작은 팀으로 약 1,500개의 pull request가 열리고 병합되었다. 이는 엔지니어 1명당 하루 평균 3.5개의 PR에 해당하며, 놀랍게도 팀이 현재 7명의 엔지니어로 성장하면서 처리량이 _증가했다_. 중요한 것은 이것이 결과물을 위한 결과물이 아니었다는 점이다. 이 제품은 내부적으로 수백 명의 사용자가 사용했고, 매일 사용하는 내부 파워 유저도 포함되어 있다.

개발 과정 내내 인간은 어떤 코드도 직접 기여하지 않았다. 이것은 팀의 핵심 철학이 되었다. **수동으로 작성한 코드는 없다.**

## 엔지니어의 역할 재정의

실제 코딩을 하지 않는 것은 **시스템, 스캐폴딩, 레버리지에 초점을 맞춘 다른 종류의 엔지니어링 작업을 도입했다.**

초기 진행은 예상보다 느렸다. Codex가 무능해서가 아니라 환경이 충분히 명시되지 않았기 때문이다. 에이전트는 높은 수준의 목표를 향해 진행하는 데 필요한 도구, 추상화, 내부 구조가 부족했다. 엔지니어링 팀의 주된 업무는 에이전트가 유용한 작업을 할 수 있도록 하는 것이 되었다.

실제로 이것은 깊이 우선(depth-first) 작업을 의미했다. 더 큰 목표를 더 작은 빌딩 블록(설계, 코드, 리뷰, 테스트 등)으로 분해하고, 에이전트에게 그 블록을 구성하도록 프롬프트하고, 이를 사용하여 더 복잡한 작업을 수행하는 것이다. 무언가 실패했을 때 해결책은 거의 "더 열심히 시도하라"가 아니었다. 진행할 수 있는 유일한 방법이 Codex가 작업을 수행하도록 하는 것이었기 때문에, 인간 엔지니어는 항상 작업에 개입하여 "어떤 기능이 누락되었고, 에이전트가 이를 이해하고 적용할 수 있도록 어떻게 만들 수 있는가?"라고 물었다.

인간은 거의 전적으로 프롬프트를 통해 시스템과 상호작용한다. 엔지니어가 작업을 설명하고, 에이전트를 실행하고, pull request를 열도록 한다. PR을 완료하기 위해 우리는 Codex에게 로컬에서 자체 변경 사항을 검토하고, 로컬 및 클라우드에서 추가적인 특정 에이전트 리뷰를 요청하고, 인간이나 에이전트가 제공한 피드백에 응답하고, 모든 에이전트 리뷰어가 만족할 때까지 루프에서 반복하도록 지시한다(실제로 이것은 [Ralph Wiggum Loop](https://ghuntley.com/loop/)다). Codex는 인간이 CLI에 복사하여 붙여넣지 않고도 컨텍스트를 수집하기 위해 표준 개발 도구(gh, 로컬 스크립트, 저장소 임베디드 스킬)를 직접 사용한다.

인간이 pull request를 검토할 수도 있지만 필수는 아니다. 시간이 지나면서 우리는 거의 모든 리뷰 노력을 에이전트 간 처리로 전환했다.

## 애플리케이션 가독성 향상

코드 처리량이 증가하면서 우리의 병목은 인간 QA 역량이 되었다. 고정된 제약이 인간의 시간과 주의력이었기 때문에, 우리는 애플리케이션 UI, 로그, 앱 메트릭 같은 것들을 Codex가 직접 읽을 수 있도록 만들어 에이전트에 더 많은 기능을 추가하는 데 주력했다.

예를 들어, 우리는 앱을 git worktree당 부팅 가능하게 만들어 Codex가 변경 사항마다 하나의 인스턴스를 시작하고 구동할 수 있도록 했다. 또한 Chrome DevTools Protocol을 에이전트 런타임에 연결하고 DOM 스냅샷, 스크린샷, 내비게이션 작업을 위한 스킬을 만들었다. 이를 통해 Codex는 버그를 재현하고, 수정을 검증하고, UI 동작에 대해 직접 추론할 수 있게 되었다.

관찰성 도구에 대해서도 동일한 작업을 수행했다. 로그, 메트릭, 트레이스는 특정 worktree에 대해 임시적인 로컬 관찰성 스택을 통해 Codex에 노출된다. Codex는 해당 앱의 완전히 격리된 버전에서 작업하며, 로그와 메트릭을 포함해 작업이 완료되면 해체된다. 에이전트는 LogQL로 로그를 쿼리하고 PromQL로 메트릭을 쿼리할 수 있다. 이러한 컨텍스트가 제공되면 "서비스 시작이 800ms 이내에 완료되도록 보장" 또는 "이 네 가지 중요한 사용자 여정에서 span이 2초를 초과하지 않도록" 같은 프롬프트가 실행 가능해진다.

우리는 정기적으로 단일 Codex 실행이 단일 작업에서 최대 6시간 동안 작업하는 것을 본다(종종 인간이 자는 동안).

## 저장소 지식을 기록 시스템으로 만들었다

컨텍스트 관리는 에이전트를 크고 복잡한 작업에서 효과적으로 만드는 가장 큰 과제 중 하나다. 우리가 배운 가장 초기의 교훈 중 하나는 간단했다. **Codex에게 1,000페이지 지침 매뉴얼이 아닌 지도를 제공하라.**

*   **컨텍스트는 희소한 자원이다.** 거대한 지침 파일은 작업, 코드, 관련 문서를 밀어낸다. 그래서 에이전트는 주요 제약을 놓치거나 잘못된 것을 최적화하기 시작한다.
*   **너무 많은 지침은** **_비지침_****이 된다.** 모든 것이 "중요"하면 아무것도 중요하지 않다. 에이전트는 의도적으로 탐색하는 대신 로컬에서 패턴 매칭을 하게 된다.
*   **즉시 썩는다.** 모놀리식 매뉴얼은 오래된 규칙의 무덤이 된다. 에이전트는 무엇이 여전히 사실인지 알 수 없고, 인간은 유지 관리를 중단하며, 파일은 조용히 매력적인 성가신 것이 된다.
*   **검증하기 어렵다.** 단일 블롭은 기계적 검사(커버리지, 신선도, 소유권, 교차 링크)에 적합하지 않으므로 드리프트는 불가피하다.

그래서 우리는 `AGENTS.md`를 백과사전으로 취급하는 대신 **목차**로 취급한다.

저장소의 지식 베이스는 기록 시스템으로 취급되는 구조화된 `docs/` 디렉토리에 존재한다. 짧은 `AGENTS.md`(약 100줄)가 컨텍스트에 주입되며 주로 지도 역할을 하고, 다른 곳의 더 깊은 진실의 원천을 가리킨다.

저장소 내 지식 저장소 레이아웃.

설계 문서는 카탈로그화되고 인덱싱되며, 검증 상태와 에이전트 우선 운영 원칙을 정의하는 핵심 신념 세트를 포함한다. [아키텍처 문서](https://matklad.github.io/2021/02/06/ARCHITECTURE.md.html)는 도메인과 패키지 계층의 최상위 지도를 제공한다. 품질 문서는 각 제품 도메인과 아키텍처 계층을 등급화하여 시간 경과에 따른 격차를 추적한다.

계획은 일급 아티팩트로 취급된다. 임시 경량 계획은 작은 변경에 사용되는 반면, 복잡한 작업은 저장소에 체크인되는 진행 및 결정 로그가 있는 [실행 계획](https://cookbook.openai.com/articles/codex_exec_plans)에 캡처된다. 활성 계획, 완료된 계획, 알려진 기술 부채는 모두 버전이 지정되고 공동 배치되어 에이전트가 외부 컨텍스트에 의존하지 않고 작동할 수 있다.

이것은 **점진적 공개**를 가능하게 한다. 에이전트는 작고 안정적인 진입점에서 시작하고 다음에 어디를 봐야 할지 배우게 되며, 처음부터 압도당하지 않는다.

우리는 이것을 기계적으로 시행한다. 전용 린터와 CI 작업은 지식 베이스가 최신이고 교차 링크되어 있으며 올바르게 구조화되어 있는지 검증한다. 반복적인 "문서 정리(doc-gardening)" 에이전트는 실제 코드 동작을 반영하지 않는 오래되거나 쓸모없는 문서를 스캔하고 수정 pull request를 연다.

## 에이전트 가독성이 목표다

코드베이스가 진화하면서 Codex의 설계 결정 프레임워크도 진화해야 했다.

저장소가 완전히 에이전트 생성이기 때문에 먼저 _Codex의_ _가독성_에 최적화된다. 팀이 신입 엔지니어를 위해 코드의 탐색성을 개선하려는 것과 같은 방식으로, 인간 엔지니어의 목표는 에이전트가 **저장소 자체에서 직접** 전체 비즈니스 도메인에 대해 추론할 수 있도록 만드는 것이었다.

에이전트의 관점에서 실행 중에 컨텍스트 내에서 액세스할 수 없는 것은 사실상 존재하지 않는다. Google Docs, 채팅 스레드, 사람들의 머릿속에 존재하는 지식은 시스템에서 액세스할 수 없다. 저장소 로컬, 버전이 지정된 아티팩트(예: 코드, markdown, 스키마, 실행 가능한 계획)만 볼 수 있다.

우리는 시간이 지남에 따라 점점 더 많은 컨텍스트를 저장소로 푸시해야 한다는 것을 배웠다. 팀을 아키텍처 패턴에 맞춘 그 Slack 토론은 어떤가? 에이전트가 발견할 수 없다면, 3개월 후에 합류한 신입 사원에게 알려지지 않은 것과 같은 방식으로 읽을 수 없다.

Codex에 더 많은 컨텍스트를 제공한다는 것은 임시 지침으로 압도하는 대신 에이전트가 그것에 대해 추론할 수 있도록 올바른 정보를 구성하고 노출하는 것을 의미한다. 신입 팀원에게 제품 원칙, 엔지니어링 규범, 팀 문화(이모지 선호도 포함)에 대해 온보딩하는 것과 같은 방식으로, 에이전트에게 이 정보를 제공하면 더 잘 정렬된 출력으로 이어진다.

이 프레이밍은 많은 트레이드오프를 명확하게 했다. 우리는 저장소 내에서 완전히 내부화되고 추론될 수 있는 종속성과 추상화를 선호했다. "지루한" 것으로 종종 설명되는 기술은 구성 가능성, API 안정성, 훈련 세트의 표현으로 인해 에이전트가 모델링하기 더 쉬운 경향이 있다. 어떤 경우에는 공개 라이브러리의 불투명한 업스트림 동작을 해결하는 것보다 에이전트가 기능의 하위 집합을 다시 구현하도록 하는 것이 더 저렴했다. 예를 들어, 일반 `p-limit` 스타일 패키지를 가져오는 대신 자체 map-with-concurrency 헬퍼를 구현했다. 이것은 OpenTelemetry 계측과 긴밀하게 통합되어 있고, 100% 테스트 커버리지를 가지며, 런타임이 예상하는 대로 정확하게 동작한다.

시스템의 더 많은 부분을 에이전트가 직접 검사, 검증, 수정할 수 있는 형태로 가져오는 것은 레버리지를 증가시킨다. Codex뿐만 아니라 코드베이스에서 작업하는 다른 에이전트(예: [Aardvark](https://openai.com/index/introducing-aardvark/))에게도 마찬가지다.

## 아키텍처와 취향 시행

문서만으로는 완전히 에이전트 생성된 코드베이스를 일관성 있게 유지할 수 없다. **구현을 세세하게 관리하지 않고 불변성을 시행함으로써, 기반을 훼손하지 않으면서 에이전트가 빠르게 배포할 수 있도록 한다.** 예를 들어, 우리는 Codex가 [경계에서 데이터 형태를 파싱](https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/)하도록 요구하지만, 그것이 어떻게 발생하는지에 대해서는 규정하지 않는다(모델은 Zod를 좋아하는 것 같지만, 우리는 그 특정 라이브러리를 지정하지 않았다).

에이전트는 [엄격한 경계와 예측 가능한 구조](https://bits.logic.inc/p/ai-is-forcing-us-to-write-good-code)가 있는 환경에서 가장 효과적이므로, 우리는 엄격한 아키텍처 모델을 중심으로 애플리케이션을 구축했다. 각 비즈니스 도메인은 고정된 레이어 세트로 나뉘며, 엄격하게 검증된 종속성 방향과 제한된 허용 가능한 에지 세트가 있다. 이러한 제약은 커스텀 린터(물론 Codex 생성!)와 구조적 테스트를 통해 기계적으로 시행된다.

아래 다이어그램은 규칙을 보여준다. 각 비즈니스 도메인(예: App Settings) 내에서 코드는 고정된 레이어 세트(Types → Config → Repo → Service → Runtime → UI)를 통해 "앞으로"만 의존할 수 있다. 횡단 관심사(auth, connectors, telemetry, feature flags)는 단일 명시적 인터페이스인 Providers를 통해 진입한다. 그 외의 것은 허용되지 않으며 기계적으로 시행된다.

이것은 보통 수백 명의 엔지니어가 있을 때까지 연기하는 종류의 아키텍처다. 코딩 에이전트와 함께라면 이것은 초기 전제 조건이다. 제약은 붕괴나 아키텍처 드리프트 없이 속도를 허용하는 것이다.

실제로 우리는 커스텀 린터와 구조적 테스트, 그리고 소수의 "취향 불변성"으로 이러한 규칙을 시행한다. 예를 들어, 우리는 구조화된 로깅, 스키마 및 타입에 대한 명명 규칙, 파일 크기 제한, 플랫폼별 안정성 요구 사항을 커스텀 린트로 정적으로 시행한다. 린트가 커스텀이기 때문에 에이전트 컨텍스트에 수정 지침을 주입하기 위해 오류 메시지를 작성한다.

인간 우선 워크플로에서 이러한 규칙은 지나치게 세세하거나 제약적으로 느껴질 수 있다. 에이전트와 함께라면 이것들은 승수가 된다. 한 번 인코딩되면 모든 곳에 한 번에 적용된다.

동시에 우리는 제약이 중요한 곳과 그렇지 않은 곳에 대해 명시적이다. 이것은 대규모 엔지니어링 플랫폼 조직을 이끄는 것과 유사하다. 경계를 중앙에서 시행하고, 로컬에서 자율성을 허용한다. 경계, 정확성, 재현 가능성에 대해 깊이 관심을 가진다. 그 경계 내에서 팀 또는 에이전트에게 솔루션이 표현되는 방식에 상당한 자유를 허용한다.

결과 코드가 항상 인간의 스타일 선호도와 일치하지는 않으며, 그것은 괜찮다. 출력이 정확하고, 유지 관리 가능하며, 미래의 에이전트 실행에 읽을 수 있는 한 기준을 충족한다.

인간의 취향은 지속적으로 시스템에 피드백된다. 리뷰 코멘트, 리팩토링 pull request, 사용자 대면 버그는 문서 업데이트로 캡처되거나 도구에 직접 인코딩된다. 문서가 부족할 때 규칙을 코드로 승격한다.

## 처리량은 병합 철학을 바꾼다

Codex의 처리량이 증가함에 따라 많은 기존 엔지니어링 규범이 역효과를 냈다.

저장소는 최소한의 차단 병합 게이트로 운영된다. Pull request는 수명이 짧다. 테스트 플레이크는 종종 진행을 무한정 차단하는 대신 후속 실행으로 해결된다. 에이전트 처리량이 인간의 주의력을 훨씬 초과하는 시스템에서 수정은 저렴하고 대기는 비싸다.

이것은 낮은 처리량 환경에서는 무책임할 것이다. 여기서는 종종 올바른 트레이드오프다.

## "에이전트 생성"이 실제로 의미하는 것

우리가 코드베이스가 Codex 에이전트에 의해 생성되었다고 말할 때, 코드베이스의 모든 것을 의미한다.

에이전트는 다음을 생성한다:

*   제품 코드 및 테스트
*   CI 구성 및 릴리스 도구
*   내부 개발자 도구
*   문서 및 설계 히스토리
*   평가 하네스
*   리뷰 코멘트 및 응답
*   저장소 자체를 관리하는 스크립트
*   프로덕션 대시보드 정의 파일

인간은 항상 루프에 남아 있지만 이전과 다른 추상화 계층에서 작업한다. 우리는 작업의 우선순위를 정하고, 사용자 피드백을 수용 기준으로 번역하고, 결과를 검증한다. 에이전트가 어려움을 겪을 때 우리는 그것을 신호로 취급한다. 무엇이 누락되었는지(도구, 가드레일, 문서) 식별하고 저장소에 다시 피드백하며, 항상 Codex 자체가 수정을 작성하도록 한다.

에이전트는 표준 개발 도구를 직접 사용한다. 리뷰 피드백을 가져오고, 인라인으로 응답하고, 업데이트를 푸시하며, 종종 자체 pull request를 squash하고 병합한다.

## 자율성 수준 증가

개발 루프의 더 많은 부분(테스트, 검증, 리뷰, 피드백 처리, 복구)이 시스템에 직접 인코딩되면서, 저장소는 최근 Codex가 엔드투엔드로 새로운 기능을 구동할 수 있는 의미 있는 임계값을 넘었다.

단일 프롬프트가 주어지면 에이전트는 이제 다음을 수행할 수 있다:

*   코드베이스의 현재 상태 검증
*   보고된 버그 재현
*   실패를 보여주는 비디오 녹화
*   수정 구현
*   애플리케이션을 구동하여 수정 검증
*   해결을 보여주는 두 번째 비디오 녹화
*   Pull request 열기
*   에이전트 및 인간 피드백에 응답
*   빌드 실패 감지 및 수정
*   판단이 필요할 때만 인간에게 에스컬레이션
*   변경 사항 병합

이 동작은 이 저장소의 특정 구조와 도구에 크게 의존하며, 적어도 아직은 유사한 투자 없이 일반화된다고 가정해서는 안 된다.

## 엔트로피와 가비지 컬렉션

**완전한 에이전트 자율성은 또한 새로운 문제를 도입한다.** Codex는 저장소에 이미 존재하는 패턴을 복제한다. 고르지 않거나 최적이 아닌 패턴도 마찬가지다. 시간이 지나면서 이것은 필연적으로 드리프트로 이어진다.

처음에는 인간이 이를 수동으로 처리했다. 우리 팀은 매주 금요일(주의 20%)을 "AI 쓰레기" 청소에 사용했다. 당연히 그것은 확장되지 않았다.

대신 우리는 "황금 원칙"이라고 부르는 것을 저장소에 직접 인코딩하고 반복적인 정리 프로세스를 구축하기 시작했다. 이러한 원칙은 미래의 에이전트 실행을 위해 코드베이스를 읽기 쉽고 일관성 있게 유지하는 의견이 반영된 기계적 규칙이다. 예를 들어: (1) 불변성을 중앙 집중화하기 위해 손으로 만든 헬퍼보다 공유 유틸리티 패키지를 선호하고, (2) "YOLO 스타일"로 데이터를 탐색하지 않는다. 경계를 검증하거나 타입이 지정된 SDK에 의존하여 에이전트가 추측된 형태로 실수로 구축할 수 없도록 한다. 정기적으로 편차를 스캔하고, 품질 등급을 업데이트하고, 타겟 리팩토링 pull request를 여는 백그라운드 Codex 작업 세트가 있다. 대부분은 1분 미만으로 검토할 수 있으며 자동 병합된다.

이것은 가비지 컬렉션처럼 기능한다. 기술 부채는 고금리 대출과 같다. 복리로 늘어나서 고통스러운 폭발로 해결하는 것보다 작은 증분으로 지속적으로 갚는 것이 거의 항상 더 낫다. 인간의 취향은 한 번 캡처된 다음 모든 코드 줄에서 지속적으로 시행된다. 이것은 또한 나쁜 패턴이 며칠 또는 몇 주 동안 코드베이스에 퍼지도록 하는 대신 매일 단위로 포착하고 해결할 수 있게 한다.

## 우리가 여전히 배우고 있는 것

이 전략은 지금까지 OpenAI의 내부 출시 및 채택을 통해 잘 작동했다. 실제 사용자를 위한 실제 제품을 구축하는 것은 우리의 투자를 현실에 고정하고 장기적인 유지 관리성을 향해 우리를 안내하는 데 도움이 되었다.

우리가 아직 모르는 것은 완전히 에이전트 생성된 시스템에서 아키텍처 일관성이 수년에 걸쳐 어떻게 진화하는가이다. 우리는 여전히 인간의 판단이 가장 많은 레버리지를 추가하는 곳과 그 판단을 복리로 증가시키기 위해 어떻게 인코딩하는지 배우고 있다. 또한 모델이 시간이 지남에 따라 계속 더 유능해짐에 따라 이 시스템이 어떻게 진화할지 모른다.

분명해진 것은 소프트웨어를 구축하는 것이 여전히 규율을 요구하지만, 규율은 코드보다 스캐폴딩에서 더 많이 나타난다는 것이다. 코드베이스를 일관성 있게 유지하는 도구, 추상화, 피드백 루프가 점점 더 중요해지고 있다.

**우리의 가장 어려운 과제는 이제 에이전트가 우리의 목표를 달성하는 데 도움이 되는 환경, 피드백 루프, 제어 시스템을 설계하는 데 집중되어 있다.** 대규모로 복잡하고 안정적인 소프트웨어를 구축하고 유지 관리하는 것.

Codex와 같은 에이전트가 소프트웨어 라이프사이클의 더 큰 부분을 맡게 됨에 따라 이러한 질문은 훨씬 더 중요해질 것이다. 우리는 몇 가지 초기 교훈을 공유하는 것이 [당신이 그냥 물건을 만들 수 있도록](https://openai.com/codex/) 노력을 어디에 투자해야 하는지 추론하는 데 도움이 되기를 바란다.

### 1. 에이전트 가독성: "볼 수 없으면 존재하지 않는다"

보통 코드 가독성이라 하면 인간 개발자가 읽기 쉬운 코드를 떠올린다. 변수명을 잘 짓고, 주석을 달고, 함수를 잘 나누는 것. 그런데 이 팀은 코드를 사람이 아니라 Codex 에이전트가 읽고 이해할 수 있게 최적화한다. 에이전트가 실제로 코드를 작성하고 수정하는 주체이기 때문이다.

핵심 원칙: **에이전트 실행 중에 컨텍스트 내에서 액세스할 수 없는 것은 사실상 존재하지 않는다.**

이것이 의미하는 바를 표로 정리하면:

| 에이전트가 볼 수 있는 것 | 에이전트가 볼 수 없는 것 |
|---|---|
| 저장소 안의 코드 | Slack에서 나눈 아키텍처 토론 |
| markdown 문서 | Google Docs에 있는 기획서 |
| 스키마 파일 | 사람 머릿속의 암묵지 |
| 실행 가능한 계획 파일 | 구두로 전달된 컨벤션 |

예를 들어, 팀이 Slack에서 "앞으로 이 패턴으로 하자"라고 합의했다 해도 저장소에 기록하지 않으면 에이전트는 모른다. 마치 신입 사원에게 알려주지 않은 암묵적 규칙과 같다. 그래서 이 팀은 모든 결정과 규칙을 저장소 안에 버전 관리되는 파일로 넣는다.

**`p-limit` 대신 자체 구현한 이유.** `p-limit`은 Node.js에서 동시 실행 수를 제한하는 인기 라이브러리다. 하지만 이 팀은 이를 안 쓰고 직접 만들었다. 외부 라이브러리를 쓰면 에이전트 입장에서 (1) 코드가 npm 패키지 안에 숨어있어서 동작 방식을 직접 읽고 추론할 수 없고, (2) 버그가 생겼을 때 내 코드 문제인지 라이브러리 문제인지 판단이 어렵고, (3) 라이브러리의 엣지 케이스를 알 수 없다. 자체 구현하면 코드가 저장소 안에 있어 에이전트가 전체 동작을 직접 읽을 수 있고, OpenTelemetry 계측과 딱 맞게 통합되며, 100% 테스트 커버리지라 동작이 예측 가능하다. 비유하면, 남의 집 블랙박스 기계를 빌려 쓰는 것 vs. 내 작업장에 투명한 기계를 직접 만들어 놓는 것의 차이다.

**"지루한 기술"을 선호하는 이유.** "boring"으로 종종 설명되는 기술이 에이전트에게 더 좋다. 이유가 세 가지다: (1) 구성 가능성(Composability) — 단순한 기술일수록 조합해서 쓰기 쉽다, (2) API 안정성 — 자주 바뀌지 않아서 에이전트가 학습한 패턴이 계속 유효하다, (3) 훈련 세트의 표현 — LLM이 학습 데이터에서 많이 본 기술이라 더 잘 다룬다. 이는 [[i-was-wrong|I was wrong]] 노트에서 다뤘던 "Agent-Friendly DX" 개념과 직접 연결된다. 타입 시스템, 테스트, 린터 같은 "지루하지만 견고한" 도구가 에이전트에게는 가장 효과적인 피드백 루프가 된다.

### 2. 아키텍처와 취향 시행: 경계는 엄격하게, 내부는 자유롭게

문서만으로는 완전히 에이전트 생성된 코드베이스를 일관성 있게 유지할 수 없다. 핵심은 **구현을 세세하게 관리하지 않고 불변성을 시행**하는 것이다. 비유하면 도시 계획에서 "이 구역은 주거용, 저 구역은 상업용"이라는 용도 규제(zoning)는 엄격하게 시행하지만, 그 안에서 건물을 어떤 스타일로 짓는지는 건축가에게 맡기는 것과 같다.

글에서 소개하는 레이어 모델:

| 레이어 | 역할 |
|---|---|
| Types | 데이터 타입 정의 |
| Config | 설정값 |
| Repo | 데이터 접근 |
| Service | 비즈니스 로직 |
| Runtime | 실행 환경 |
| UI | 사용자 인터페이스 |

의존성은 무조건 "앞으로"만 가능하다. Types -> Config -> Repo -> Service -> Runtime -> UI 순서로만 의존할 수 있고, 역방향은 커스텀 린터로 기계적으로 금지된다.

**"Parse, Don't Validate" 원칙.** 외부에서 데이터가 들어올 때 "형태가 맞는지 확인하고 넘기기"가 아니라, "파싱해서 올바른 타입으로 변환한 후에야 시스템 안으로 들여보내기"를 의미한다. 재미있는 점은 팀이 "Zod로 파싱하라"고 지정하지 않았다는 것이다. "경계에서 파싱하라"는 원칙(what)만 시행하고, 어떤 도구를 쓸지(how)는 에이전트에게 맡겼다. 결과적으로 Codex가 Zod를 자주 선택했지만, 그건 에이전트의 자율적 결정이었다.

**취향 불변성.** 아키텍처 규칙 외에도 구조화된 로깅, 명명 규칙, 파일 크기 제한, 플랫폼별 안정성 요구사항 같은 "취향"도 커스텀 린트로 시행한다. 인간 팀에서는 "너무 세세하다"고 느껴질 수 있지만, 에이전트에게는 한 번 인코딩하면 모든 PR에 자동으로 적용되는 승수(multiplier) 역할을 한다. 그리고 린트 오류 메시지 자체에 수정 지침을 넣어서, 에이전트가 오류를 보면 바로 문서를 참고해서 고칠 수 있게 한다. 보통 린트 메시지는 사람에게 "이거 고쳐"라고 알려주는 용도인데, 여기선 에이전트에게 수정 방법까지 주입하는 채널로 쓴다.

**왜 이게 초기 전제 조건인가.** 인간 팀 3-5명에서 이렇게 엄격한 아키텍처를 세우면 오버엔지니어링이다. 사람은 암묵적으로 규칙을 공유하고 코드 리뷰에서 잡아내니까. 하지만 에이전트는 암묵지가 없다. 명시적 규칙이 없으면 각 에이전트 실행이 제각각 다른 패턴으로 코드를 만들어서 금방 혼란스러워진다. [[effective-harnesses-for-long-running-agents-ko|장시간 실행 에이전트를 위한 효과적인 하네스 구축]]에서도 같은 원리가 나온다.

### 핵심 인사이트

> 에이전트 시대의 아키텍처는 "인간이 읽기 좋은 코드"가 아니라 "에이전트가 추론할 수 있는 환경"을 설계하는 것이다. 모든 결정과 규칙은 저장소 안에 버전 관리되는 형태로 존재해야 하며, "경계는 엄격하게 시행하되 내부 구현은 자유롭게" 두는 것이 에이전트의 속도와 코드베이스의 일관성을 동시에 달성하는 핵심 전략이다.
